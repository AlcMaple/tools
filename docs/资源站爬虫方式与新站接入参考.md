## 0. 三站速览对照

| 维度 | 喜番 Xifan | Girigiri | Aowu (FantasyKon) |
|------|-----------|----------|-------------------|
| **技术栈** | MacCMS dsn2 模板 | MacCMS dsn2 模板 | 自研 SPA + 加密 REST API（2026-05 从 MacCMS 改版） |
| **数据获取** | HTML 抓取（cheerio） | HTML 抓取（cheerio） | 加密 API（`POST /api/site/secure`，AES-256-GCM 信封） |
| **搜索** | `GET /search.html?wd=` | `GET /search/-------------/?wd=` | `callSecure(bundle:search)`，服务端分页 |
| **分页** | 共享 `crawlAllPages`（跟「下一页」链接） | 同左 | 服务端 `total/limit`，前端封顶 10 页 |
| **剧集列表** | 详情/播放页 HTML 解析 | `/GV{id}/` 列表页 + `/playGV{id}-{源}-{集}/` 播放页 | `bundle:video` → `sources[].episodes[]` |
| **视频地址** | **MP4 直链**（播放页 `var player_aaaa` JSON 里 `url`，需 `decodeURIComponent`） | **HLS m3u8**（隐藏 BrowserWindow 拦截 `.m3u8` 请求） | **签名 MP4**（两步 `bundle:play`→`play`，落字节系 CDN） |
| **下载器** | `mp4-range-downloader`，8 线程 | TS 段并发(8) + AES 解密 + ffmpeg 合并 | `mp4-range-downloader`，**单线程**（CDN 限并发） |
| **断点续传** | `.partN` 分块续传 | 跳过已存在 TS 段 | 单流 Range 续传 |
| **反爬重点** | 图形验证码 + cookie 会话 | 图形验证码 + cookie 会话 + TS 可能 AES-128-CBC | 首页派生密钥的加密信封 + 节流 |
| **加密** | 无（URL 仅 encode） | TS 分段可能 AES-128-CBC（m3u8 带 `#EXT-X-KEY`） | AES-256-GCM 全程 + 密钥派生 |

**一句话归纳三类站**：
1. **MacCMS HTML 直链站**（Xifan）—— 最简单，正则抠 `player_aaaa` 拿 MP4。
2. **MacCMS HTML + HLS 站**（Girigiri）—— 视频走 m3u8，用浏览器窗口拦截真实地址，TS 段下载 + ffmpeg 合并。
3. **自研加密 API 站**（Aowu）—— 要逆向加密信封 + 密钥派生，工作量最大，单独有手册。

## 1. 不可逾越的红线（先记牢，再写代码）

这些是项目级红线，违反会**实际伤害用户**，完整 rationale 见 `docs/bgm-集成参考手册.md` 与根目录 `CLAUDE.md`：

1. **所有主进程 HTTP 必须走 `src/main/shared/net-request.ts` 的 `netRequest()`（Electron `net`）**，禁止 Node `https`/`axios`/`node-fetch`。
   - 原因：Node `https` **不读系统代理**。用户开 Clash/Mihomo 的 fake-ip 代理时，DNS 把域名解析成 `198.18.x` 不可路由假地址 → Node 直连黑洞 → 超时。Electron `net` 走 Chromium 网络栈，自动跟系统代理/PAC，行为与浏览器一致。
   - 例外：用隐藏 `BrowserWindow` 抓 m3u8（Girigiri）本身就是 Chromium，天然走代理，OK。
2. **失败后不重试、不试探**。限流/5xx 一律 `throw` 到 UI，由用户看倒计时按钮自己决定何时重试。
   - 禁止：应用层自动重试、周期性「探活」、`catch → return null` 静默吞错、IP 池/代理轮换/Playwright「绕过」限流（全部曾被否决）。
   - 唯一允许的代码级重试：传输层瞬断（`withTransientRetry`，单次 ECONNRESET 级 socket 抖动重试）。
3. **UA 按目标区分**：诚实 API（如 `api.bgm.tv`）用 `MapleTools/<ver>`；HTML 抓取用 `BrowserSession` 的随机浏览器 UA。别混用。
4. 外部请求统一过 `shared/rate-limit.ts` 的 `RateLimiter`（间隔 + 抖动）串行节流。

## 2. 可复用工具箱（`src/main/shared/`）

接新站之前先看这层 —— 大部分轮子已经造好，新站通常只需写一个 `api.ts` + IPC 注册。

| 模块 | 作用 | 谁在用 |
|------|------|--------|
| **`net-request.ts`** | `netRequest(url, opts)` —— 走 Chromium 网络栈的 HTTP，自动跟系统代理、自动解压、默认跟 3xx、12s 超时。**剔除调用方的 `Accept-Encoding`/`Host`/`Connection`/`Content-Length`**（这些由 net 自管，手动设会 `ERR_INVALID_ARGUMENT`）。不接节流、不重试，纯传输层。 | 全部 |
| **`browser-session.ts`** | `BrowserSession` —— 每站一个实例：从 5 个 Chrome 大版本(119-123)随机选一个，**整个 app 生命周期固定不变**（真浏览器不中途换 UA），UA + `sec-ch-ua` + `sec-ch-ua-platform` 同版本对齐；内置 cookie jar（`ingestSetCookie` 吃 `Set-Cookie`，`headers()` 自动回带 `Cookie`）；`headers(extra)` 一把生成完整浏览器头（含 `sec-fetch-*`、Referer）。 | Aowu、BGM HTML |
| **`http-session.ts`** | `HttpSession` —— 更轻的 cookie 会话：cookie 持久化到 `userData/.{name}_cookies.json`，自动跟 301/302。验证码通过后续会话靠它保持登录态。 | Xifan、Girigiri |
| **`rate-limit.ts`** | `RateLimiter`（硬性最小间隔 + 抖动 + 滚动窗口配额 + 软节流恢复）、`LimitDetector`（嗅探 HTTP 200 但 body 是限流页，如「N 秒内只能搜索一次」）、`RateLimitError`（带 `retryAfterSec`，**不重试直接抛**给 UI 倒计时）。 | 全部外部站 |
| **`maccms-search-paginator.ts`** | `crawlAllPages({firstHtml, baseUrl, parsePage, fetchHtml})` —— MacCMS dsn2 通用分页：**跟「下一页」链接**而非算 URL（各站 search path 不同），顺序请求、每页默认 1s 间隔、封顶 20 页、空页/自环自动停。 | Xifan、Girigiri、（旧 Aowu） |
| **`mp4-range-downloader.ts`** | `downloadByUrl(url, savePath, signal, onProgress, {threadCount})` —— MP4 多线程 Range 下载：HEAD 探测 206→多线程(默认 8)/200→单流降级；`.partN` 分块、断点续传、自动跟 CDN 302、每块重试 5 次指数退避、最后合并。 | Xifan、Aowu |
| **`site-download-queue.ts` / `download-scheduler.ts`** | 每站一个内存队列：`pending[]` + `priorityFront[]`（插队续传某集）+ `pausedEps` + 当前集 `AbortController`，串行逐集下载。支持暂停/恢复/重排。**重启丢失**，靠 renderer 的 localStorage 重建。 | 全部下载源 |
| **`download-types.ts`** | `DlEvent`（进度事件）、`safeName()`（文件名清洗）、`DESKTOP_USER_AGENT`（下载用的固定桌面 UA）。所有下载源在统一频道 `download:progress` 发进度。 | 全部 |

## 3. 三类站点的具体打法

### A. MacCMS HTML 直链站 —— 以 Xifan 为模板

**识别**：搜索结果页有 `<a class="page-link">` 分页块；播放页里有 `var player_aaaa = {...}`。这是苹果 CMS(maccms) dsn2 模板的指纹。

**抓取链路**：
```
search.html?wd={kw}
  → cheerio 解析卡片：h3.slide-info-title(标题/详情链)、
    div.vod-detail-bnt a.button(播放链)、div.detail-pic img@data-src(封面，注意是 data-src)
  → crawlAllPages 跟「下一页」翻完所有页
播放页 watch
  → 正则抠 var player_aaaa（兼容紧凑 JSON 和带换行的分块两种格式）
  → 取 .url 字段 → decodeURIComponent → MP4 直链
  → 多源：div.anthology-tab a.vod-playerUrl 列出各源；徽章 span.badge 读总集数
集数 URL 构造
  → 访问 /watch/{id}/{源}/1.html 拿第 1 集真实 URL
  → buildTemplate：正则把末尾数字换成 {:02d} 占位符
    (例 http://x/01.mp4 → http://x/{:02d}.mp4)
  → 后续集 = 模板填充集号
下载
  → mp4-range-downloader 8 线程；切源时清掉旧 .partN / .mp4（字节不可复用）
```

**反爬**：搜索可能弹图形验证码 —— `needsCaptcha(html)` 嗅 `name="verify"`/`ds-verify-img`；`getCaptcha()` 取图给 UI、`verifyCaptcha(code)` 回提交，过了之后 cookie 保持会话。无加密。

参考源码：`src/main/xifan/api.ts`、`xifan/download.ts`。

### B. MacCMS HTML + HLS 站 —— 以 Girigiri 为模板

与 A 的区别只在**视频地址**：不是 HTML 里的明文 MP4，而是**播放器运行时才请求的 m3u8**。HTML 抓不到，所以用浏览器拦截。

**抓取链路**：
```
搜索/剧集列表：同 MacCMS（cheerio + crawlAllPages）
  列表页 /GV{id}/ 出片源 tab(a.vod-playerUrl@data-form: cht繁/chs简) + 各源集数列表
  播放页 /playGV{id}-{源idx}-{集idx}/（tab 动态渲染失败时回播放页反爬源名）
拿真实 m3u8：
  → 开隐藏 BrowserWindow 加载播放页（带 cookie）
  → session.webRequest.onBeforeRequest 拦截，过滤 pathname 以 .m3u8 结尾且 host 含 girigirilove
  → 30s 超时，返回第一个命中的 m3u8 URL
下载（HLS）：
  → 解析 m3u8：#EXTM3U 校验；#EXT-X-KEY 取 AES-128 密钥 URI + IV；非注释行=TS 段
  → 临时目录 temp/girigiri_ts/{title}_{ep}/，文件名 segment_{05d}.ts
  → 并发 8 下 TS，每段重试 8 次指数退避，跳过已存在段(断点续传)，AbortSignal 秒级取消
  → 若加密：crypto.createDecipheriv('aes-128-cbc', key, iv) 逐段解密
  → ffmpeg -f concat -safe 0 -i segments.txt -c copy -bsf:a aac_adtstoasc -movflags +faststart 合并成 mp4
  → SESSION_ID(每次启动新 UUID)隔离临时目录，防旧 app 损坏段被复用
```

**反爬**：同 A 的验证码 + cookie；额外是 TS 段可能 AES-128-CBC 加密（密钥从 m3u8 的 `#EXT-X-KEY` URI 现取）。

参考源码：`src/main/girigiri/api.ts`、`girigiri/download.ts`、`shared/http-session.ts`。

### C. 自研加密 API 站 —— 以 Aowu(FantasyKon) 为模板

**识别**：首页 GET 是 HTML 壳，但所有数据操作都走单一加密入口 `POST /api/site/secure`，请求和响应都是 `{ n, d }` 信封（无明文 JSON API）。

**加密信封（AES-256-GCM）**：
```
信封格式：{ "n": base64(12字节随机IV), "d": base64(密文 + 16字节GCM认证tag) }
加密：createCipheriv('aes-256-gcm', key, iv)，明文是 JSON.stringify(payload)
解密：createDecipheriv，校验 GCM tag
```

**密钥派生（最关键）**：32 字节 AES 密钥从**首页 HTML 的 5 个碎片**拼出来：
```
parts = gt(meta[name="fk-p"]@content)      // 9 字符
      + gt(html[data-fk-s])                // 9 字符
      + gt(window.__FKM[0])                // 9 字符
      + gt(CSS html 变量 --fk-c)           // 9 字符
      + gt(window.__FKM[1])                // 8 字符（含 padding）
其中 gt(s) = 去首尾空白和引号；parts 共 44 字符 base64
key = Buffer.from(parts,'base64').toString('latin1') 再按 utf-8 取字节 → 32 字节
（当前所有字节 ≤0x7F，latin1→utf-8 是 identity；若将来出现 >0x7F 字节，每字节会扩成 2 字节，需复刻浏览器的 atob→TextEncoder 链路）
```

**抓取链路**：
```
bootstrapKey()：GET / 提取 5 段、derive key（缓存）
search：callSecure(bundle:search, {anime:kw, page})；响应 {list[10],page,limit,total}；封顶 10 页
watch：/v/{id} 取 tail（非数字 token 走 route() 转 id）→ callSecure(bundle:video,{id})
       → sources[].episodes[]（每集 id/no/name，无名补「第N话」）
播放地址（两步）：
  bundle:play  → 拿 play_token.token（服务端签发，客户端只透传，不需复刻其 HMAC 结构）
  play(token)  → 拿字节系 CDN 签名 MP4（lf*-imcloud-file-sign.bytetos.com，支持 Range，有效期数小时）
  url-resolver.ts 带 24h 持久化缓存 + 并发合并 + 403 失效自动 invalidate
下载：mp4-range-downloader 单线程（ByteDance 签名 CDN 限并发，多线程反而互卡「卡 97%」）
```

**请求/响应周期容错**（`callSecure`）：429/503 → 不重试直接抛限流；401/403 或解密失败 → 清 key 缓存重试一次（应对密钥轮换）；正常 200 → 解信封校验 `code===200`。

**错误分类**：`AOWU_RATE_LIMITED`(限流，等)、`AOWU_UNREACHABLE`(网络/5xx)、`AOWU_STRUCTURE_CHANGED`(**协议变了，要重逆向**)。

参考源码：`src/main/aowu/secure.ts`(加密核心)、`aowu/api.ts`、`aowu/url-resolver.ts`、`aowu/download.ts`，完整 playbook 见 `docs/aowu-FantasyKon-逆向与反爬手册.md`。

## 4. 接入一个新站的标准流程

1. **踩点判类型**：浏览器 DevTools 看 —— 搜索/列表数据是页面里直出 HTML，还是 XHR/fetch 拿 JSON？视频地址是 `<video src>`/页面变量，还是播放器运行时请求的 m3u8/接口？
   - 直出 HTML + 有 `page-link` 分页 → **A 类（MacCMS）**，最省事。
   - HTML 但视频是 m3u8 → **B 类**。
   - 数据走加密 XHR（body 看不懂、是 base64 信封）→ **C 类**，准备逆向。
2. **建站点目录** `src/main/{newsite}/`，至少一个 `api.ts`。
3. **HTTP 一律用 `netRequest`**；HTML 抓取配 `BrowserSession` 或 `HttpSession`（要 cookie/验证码就用后者）。
4. **复用共享层**：
   - A/B 类搜索分页 → `crawlAllPages`，自己只写 `parsePage`（cheerio 选择器）。
   - MP4 下载 → `mp4-range-downloader`；HLS → 参考 Girigiri 的「BrowserWindow 拦截 m3u8 + TS 下载 + ffmpeg 合并」。
   - 下载队列 → `site-download-queue`；进度统一发 `download:progress`。
5. **节流接 `RateLimiter`**，限流嗅探接 `LimitDetector`，失败抛 `RateLimitError` 给 UI 倒计时 —— **不要自动重试**。
6. **IPC 四步**（见 `CLAUDE.md`）：main 注册 `registerNewsiteIpc()` → 接进 `registerAllIpc()` → preload 暴露 `window.newsiteApi` → `env.d.ts` 声明类型。新源进 `stores/siteApi.ts` 统一抽象。
7. **写一份本站的 debug/逆向笔记**进 `docs/`，给未来的自己留线索。

## 5. 逆向加密 API 站的 playbook（C 类浓缩版）

完整版在 `docs/aowu-FantasyKon-逆向与反爬手册.md`，这里是骨架：

1. **确认是协议变了**而非网络问题：`curl -sI` 首页应返回 200。
2. **抓真实流量**：用 Chrome MCP 注入 `window.fetch` hook，截获加密入口的 reqBody/resBody（明文 payload + 密文信封一一对应）。
3. **在浏览器控制台复现密钥派生**：照搬「5 段拼接 → atob → TextEncoder」，用 `crypto.subtle.decrypt` 解一条截获的响应验证 key 对不对。
4. **key 派生逆不出来就挖 JS bundle**：`curl` 下主 bundle，`grep "subtle|importKey|AES-GCM"` 定位，用字符串数组还原混淆代码，重点找算法字面量(`AES-GCM`/`AES-CBC`)、`importKey` 的 key 来源、key 由哪些 DOM 元素拼。
5. **纯 Node 端独立验证整条流程**（GET 首页→derive→encrypt→POST→decrypt→跑 search/detail/play），跑通再改进项目代码。
6. **改 `secure.ts`(派生/算法) + `api.ts`(字段) + `url-resolver.ts`(播放流程)**，提交。

**工时预估**：仅字段改名 1-2h；密钥派生换片段位置 3-5h；整套加密算法换(GCM→CBC+HMAC 之类) 0.5-1d；遇到 JS 挑战/TLS 指纹/Bot 管理 1-3d（可能要回退到隐藏 BrowserWindow 方案）。

**故意不做的**（成本/收益不划算，单用户量级没必要）：TLS JA3 指纹模仿、强上 HTTP/2、模拟前端埋点事件、密钥定时刷新（401/403/解密失败单次重试已兜底）。
