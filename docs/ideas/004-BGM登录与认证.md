# 004 - BGM 登录与认证

> 开始：2026-05-18
> **状态：未启动即归档（No Go）**
> **归档日期：2026-05-18**

## 🚫 为什么这阶段没做

立此存照，避免未来重新踩同一个坑。

### 决策依据

1. **BGM 没有 Personal Access Token 机制** —— bgm.tv/dev/app 只能「创建应用」
   然后跑 OAuth 2.0 dance，没有"点一下生成 token"的快捷路径。详细：必须
   填应用名 / 主页地址 / 类型 / 简介 → 拿到 App ID + App Secret → 在
   Electron 里开 BrowserWindow 走 `/oauth/authorize` 用户授权 → 回调拿
   code → POST `/oauth/access_token` 换 access_token + refresh_token（7 天
   过期，refresh_token 续期）。**实现复杂度 ≈ 一周工作量**
2. **收益完全没经过验证** —— BGM 官方文档**没有任何明确条款**说"鉴权后
   限速更宽松"。这是社区里的口口相传，没有可验证来源。曾考虑写脚本压测
   匿名 vs 带 token 对比，但脚本本身就会显著加重 IP 被 BGM 标记的程度,
   验证成本太高
3. **痛点已经被 003 解决得差不多了** —— 003 之后正常使用基本不限流。
   004 优化的是「共享 NAT / 公司网 / 多人同 IP」这种少数极端场景，把它
   做完也只是「理论上更好」
4. **App Secret 进 bundle** —— 开源 Electron 应用做 OAuth 必然要把 secret
   打到产物里（PKCE 不需要 secret，但不确定 BGM 支不支持）。VS Code 等
   开源应用是这么做的，行业惯例不算 deal breaker，但增加了运维复杂度

### 如果将来要重启这个 idea

触发条件应该是：

- **有真实用户反馈**说他们经常被限流（不是猜测）
- 并且有方法**实测验证**鉴权后限流阈值真的不一样（比如借朋友账号短期对比）
- 复杂度评估时记得 OAuth + 7 天续期 + 401 降级 + Settings UI + 文档全套

不要再因为「之前的会话里说过登录后问题少」这种**未经验证的传闻**重启
这个 idea —— 那句话出处不明，无法证伪也无法证实。

### 同一时间的诊断

归档时用户 IP 已经在 BGM 滑动惩罚窗口里（频繁触发限流 + 反复 timeout）。
解决路径是「停 24-48 小时让窗口衰减 / 切网络换 IP」，**不是**鉴权。这条
也证实了 004 解决的不是当前真实痛点。
