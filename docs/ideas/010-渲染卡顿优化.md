## 📖 详细说明

### localStorage 同步写盘下沉到 idle
**目标效果**：任何追番 / 推荐的增删改，UI 立刻响应，序列化 + 写盘不再阻塞
渲染线程；连续多次改动合并成一次写。

**复现**：追番条目较多（上百）时，编辑任意一条的状态 / 集数，能感到一下顿挫；
切 tab / 进设置返回同理。Windows 放大 2-3x。

**已经做过的事**（避免重新踩坑）：
- 根因：`animeTrackStore` / `recommendationStore` 的 `persist()` 每次 mutate
  都同步 `JSON.stringify(整张表)` + `localStorage.setItem`，再通知订阅者 ——
  追番上百条时单次 stringify 几百 KB，全跑在渲染线程上。
- 落地：新增 `src/renderer/src/utils/deferredStorage.ts`，`requestIdleCallback`
  合并写盘（同 key 多次 schedule 只留最后一个闭包，闭包在 flush 时才读 store
  抓最新值），`pagehide` / `beforeunload` / `visibilitychange=hidden` 兜底 flush
  防丢数据。两个 store 的 `persist()` 改成**先同步 notify 订阅者（UI 立即更新）
  再 schedule 写盘**。
- **不要回退**成"每次 mutate 同步写 localStorage" —— 那就是这次要消除的卡顿源。
- 延后写盘的前提：没有任何代码在 mutate 后**立即同步回读** localStorage 那个 key。
  现状满足：store 只在首次 `ensure()` 读一次，之后走内存缓存；WebDAV push 用的
  是内存里的 `tracks`，不读 localStorage。以后若新增"写完马上读盘"的逻辑要重新评估。
- downloadStore 没动 —— 它本来就有 raf 节流 + 进度事件跳过 persist，不是瓶颈。

### AnimeSyncBar dirty 检测防抖
**目标效果**：MyAnime 顶部「未同步」chip 的状态计算不再每次 store 通知都全量
stringify 整张表，列表交互不被它拖卡。

**已经做过的事**（避免重新踩坑）：
- 原本 `const currentSnapshot = useMemo(() => snapshotOf(tracks, recommendations), [...])`
  —— 但 store hook 每次通知都返回**新数组引用**，useMemo 依赖每次都变、memo 根本
  不命中，于是每次 mutate / 重渲染都同步 stringify 整张表（几百 KB）。
- 改成 200ms 防抖的 `useEffect`，结果写进 `localDirty` state。`localDirty` 只驱动
  chip 的启用态，延迟 200ms 出结果对用户无感。
- 真正 push / pull 时仍直接 `snapshotOf(...)` 算（用户主动触发，不是热路径，不用动）。

### MyAnime TrackRow memo + content-visibility
**目标效果**：单条追番编辑 / 搜索框输入时不再整列重渲染；初次进入 MyAnime 的
首屏排版只算可见的几行，上百条追番也能秒开。

**已经做过的事**（避免重新踩坑）：
- `TrackRow` 用 `memo()` 包裹：它只依赖 `track` 这一个 prop，且 `upsert` 只替换
  被改那一条 track 的对象引用（其他 key 对象身份不变），`filtered` 又是纯
  `.filter/.reverse/.sort` 不克隆元素 —— 所以单条编辑 / 搜索触发 MyAnime 重渲染时，
  未变行的 prop 引用不变，memo 跳过，重渲染从 O(N) 降到 O(1)。
- 行根 div 加 `content-visibility: auto` + `contain-intrinsic-size: auto 140px`：
  浏览器跳过视口外行的 layout/paint，首屏排版开销大降；intrinsic-size 给占位高度
  （≈卡片下限 140）保持滚动条 / 滚动位置稳定。
- **注意**：content-visibility 只省浏览器侧 layout/paint，React 仍会为全部行建 DOM
  节点（reconciliation 成本还在）。所以它和 memo 是两件事、互补：memo 省 JS 重渲染，
  content-visibility 省浏览器排版。
- **没有**上虚拟列表 / 窗口化 —— 用户明确要求别过度优化堆复杂度。这些行是变高的复杂
  卡片，虚拟化需要处理动态高度 + 滚动锚定，成本高、易出 bug。先用上面两条轻量手段，
  Windows 实测仍明显卡再考虑（见 TODO 末条），且要先测后做、别预先动手。
