# 007 - 自动更新国内加速

> 开始：2026-05-26

## 🎯 这个阶段的产物

无魔法的国内用户也能用上自动更新：**Windows** 启动 / 手动检查能在无代理网络下
连上、走国内可达镜像下载新安装包、提示重启安装；**macOS** 能查到新版本并给一个
国内加速的 dmg 下载直链（未签名仍需手动拖入 Applications）。有魔法 / 有代理的用户
可在设置里切「直连 GitHub」。整套零成本、不实名、不备案。

## 📖 详细说明

### 国内加速核心方案

**目标效果**：无魔法 Windows 用户「检查更新」能跑完整流程（查版本→下载→装），
不再死在拉 feed 文件那一步。

**已经做过的事**（避免重新踩坑）：
- 0a73903 落地：两步走 = ①读 repo 根 `update-manifest.json`（`{version,proxies}`）
  查最新版本 ②拼**固定 tag** 的 ghproxy URL 交给 electron-updater generic provider
  逐代理尝试、失败回退直连 GitHub。机制详档 `docs/自动更新-国内加速.md`。
- **被否的方案，别再走**：
  - Cloudflare —— 国内被墙/不稳，feed 拉不到，上一轮已翻车（用户明确点名这个坑）
  - 阿里云 OSS / 腾讯云 COS —— 要实名 + 费用，用户不接受
  - 自建 VPS —— 要备案，太重
  - 123pan 直链（Alas 用的）—— 要 VIP、有防盗链/流量配额，且不贴合 electron-updater
  - Gitee —— 开源仓要审核、下载偶尔要登录/限流，脆
  - 复刻 Alas 的「exe 弹控制台 git pull 源码」模式 —— Alas 是源码分发，我们是
    编译后的 Electron 安装包，git-pull 模式完全错配（用户问过，已解释否掉）
- **ghproxy 的硬约束**（实测，决定了为什么必须两步走）：
  - `/releases/latest/` 重定向 → 502（不能让更新器自己找最新版）
  - `releases.atom` → 403（不能靠 atom 查版本）
  - 固定 tag 产物、`raw.githubusercontent.com` 小文件 → 200 ✅
- electron-updater **6.8.3** 的 `newUrlFromBase` 用原生 `new URL(path, base)`，实测嵌套
  `https://` 会被完整保留（`//` 不被吞）→ generic provider 喂 ghproxy 前缀 base 成立。
- **proxies 列表写在 `update-manifest.json` 里，可远程改、不用发版** —— 这是应对
  「ghproxy 域名经常轮换」的关键设计；代码里只写死 `BOOTSTRAP_PROXIES`（拉 manifest
  本身用，有 jsdelivr + 直连兜底）。
- 稳定性取舍：靠冗余（多代理 + 远程可改列表 + 直连兜底），不靠某单点很稳。
- 发版多一步：bump 版本后必须 `npm run sync:manifest` 同步 manifest.version，否则走
  国内源的用户收不到更新（已写进 AGENTS.md §11.3）。

### 真机无魔法验证

**目标效果**：在一台无任何代理的国内网络设备上，「检查更新」能连上 GitHub main 的
manifest 并比对版本，至少显示「已是最新」（app 0.4.0 == manifest 0.4.0）。

**复现**（设备上）：
1. `git pull` → `npm run build:win` 打包
2. 运行 → 设置 → 关于 → 检查更新
3. 预期「已是最新版本」。若「无法获取更新信息」= 三通道在该网络全挂，需换引导代理。
4. 测下载：临时把 `package.json` 版本改 `0.3.0` 再打包，会真去下已发布的 v0.4.0。

**已经做过的事**：
- push 后已在本机 curl 验证三通道（ghproxy-raw / jsdelivr / 直连 raw）都 200、内容正确。
- 不能用「新打 tag」测下载 —— Actions 出的是 draft，draft 资产不公开 ghproxy 下不到；
  要用已发布的 v0.4.0。
