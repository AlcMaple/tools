## 📖 详细说明

### A. 数据层

**目标效果**：每个 `AnimeTrack` 知道自己是动画还是漫画还是小说，未来扩展
更多类目（比如游戏、三次元）有现成的字段位。

**已经做过的事**（避免重新踩坑）：

- **`subjectType` 枚举设计**：`'anime' | 'manga' | 'novel' | 'other'`。
  `other` 给 BGM 「画集 / 其他」用 —— 数据按原始 platform 值存，但 UI 只显
  动画/漫画/小说 3 个 tab，`other` 不出现在任何 tab。用户决策见 005 起始的
  AskUserQuestion 记录
- **`BgmDetail` 加 `type` 字段**：BGM API 的 `subject.type` 是数字（1=书籍 /
  2=动画 / 3=音乐 / 4=游戏 / 6=三次元）。光看 `platform` 字符串不够 ——
  动画的 platform 有 TV/剧场版/OVA 等多个值，加 `type` 后判断更稳健
- **subjectType 推导**：
  - `type === 2` → `anime`
  - `type === 1` && `platform === '漫画'` → `manga`
  - `type === 1` && `platform === '小说'` → `novel`
  - `type === 1` && 其他 → `other`
  - 其他 type（音乐/游戏等）→ `other`（实测我们不会 BGM 搜索这些类目）
- **零手动迁移**：normalize() 缺字段时默认 `'anime'`，向后兼容 + WebDAV
  同步零 schema bump（加性变更）

### B. 搜索基础设施

**目标效果**：所有 BGM 搜索路径（AnimeInfo / LinkTrackModal）都能选 cat,
默认 cat=2 保持现有调用方零改动。

**已经做过的事**（避免重新踩坑）：

- BGM URL 结构已验证（WebFetch + 用户确认）：`https://bgm.tv/subject_search/{keyword}?cat={cat}`，cat=1 是书籍（漫画+小说+画集合并），cat=2 是动画
- BGM 在 URL 层级**不支持**漫画/小说细分 —— 这是它的限制，不是我们偷懒
- IPC 签名向后兼容：`bgm:search(keyword, update?, cat?)`，cat 默认 2
- 主进程 search.ts 的限速 / 浏览器指纹 / detectLimit 都不变，只参数化 URL

### C. 搜索 UI

**目标效果**：用户搜「巨虫列岛」时能选这次搜动画还是搜漫画小说。视觉
上像 Search & Download 顶部那种"选源"风格的分段控件。

**已经做过的事**（避免重新踩坑）：

- **不要做"全搜"按钮**：用户明确说不要合并（"防止处理太多而导致限流"）。
  动画搜一次 + 书籍搜一次 = 用户两次主动操作，不要自动并发两个请求
- **缓存按 cat 隔离**：动画的缓存和书籍的缓存物理分开，否则同名作品（比如
  原作小说和动画同名）会互相覆盖。见 TODO E

### D. MyAnime tab

**目标效果**：4 个平级顶部 tab：动画 / 漫画 / 小说 / 推荐。每个类目 tab
内部跟现在动画 tab 完全一样（status chips / 排序 / 搜索 / 卡片）。

**已经做过的事**（避免重新踩坑）：

- 旧的「追番列表 / 推荐」二选一拆成 4 平级。**不**做嵌套 tab（不要"追番列表 →
  动画/漫画/小说"两层），用户决策见 AskUserQuestion 记录
- 状态过滤（全部/在追/想看/观望/看完）每个类目独立计数显示
- 「类型」「添加日期」「🌟 数」三个排序选项每个类目独立保存

### E. 缓存隔离

**目标效果**：动画搜索缓存和书籍搜索缓存物理分开 —— 同名作品（动画版 +
漫画原作）不会因为同一个 keyword 互相覆盖。

**已经做过的事**（避免重新踩坑）：

- 几种实现路径：
  1. 在 keyword 前缀加 cat：`cat2:巨虫列岛` / `cat1:巨虫列岛`
  2. 加新的 `Source` 类型 `BgmBook`，单独走一份 `search_cache_bgmbook`
- 倾向方案 1，改动小，且语义上"BGM 还是同一个源，只是细分了"。方案 2
  污染共享的 Source 抽象
