### 播放页页面设计
**目标效果**：一个播放页承载全部源——顶部 16:9 播放区，下方标题/集信息、
源切换器、集数网格。切源/切集不离开页面；失败态在播放区内展示 ErrorPanel。

**已经做过的事**（避免重新讨论）：
- 用户定调：在线观看是**独立一档功能**——原「在线观看」行的源 chips 点击
  外跳浏览器的行为不动，新增一个「在线观看」按钮作为进应用内播放页的唯一入口。
- 用户定调：嵌的是「网页里的视频组件」，**不是整个网页**——能解析出流地址的
  源用自己的播放器播，只有实在拿不到流的源才考虑嵌站点播放器组件。
- 用户定调：源切换器 = 三个默认源（稀饭/Girigiri/嗷呜）+ 该番 bindings 里的
  自定义源（如 B 站），同列同级可切换。
- 用户定调：设计**从简，拒绝过度设计**——必要元素只有四个：动漫标题、集数
  列表、当前网站内的线路选择、多个网站间的切换。不加推荐位/评论区/简介等
  任何多余模块。
- 用户定调：播放页是**沉浸式**的，不显示全局侧边栏（App.tsx 的 isFullscreen

- `docs/design-mockups/在线播放.html` 是多站需求出现前的旧稿，整体布局和

  两级：网站切换（稀饭/Girigiri/嗷呜/自定义）+ 站内线路，不能照搬。

### 站点流地址解析能力（2026-07-06 调研结论，别重新调研）
- **Xifan**：mp4 直链。URL 模板拼接，特殊集 404 时回源播放页 HTML 抽
  `player_aaaa.url`（`xifan/api.ts` `resolveEpRealUrl()`）。带 Referer 防盗链，
  无 cookie 依赖，直链长期有效。
- **Aowu**：mp4。经 `/api/site/secure` AES 加密端点两步换出字节 CDN 签名链，
  **24h TTL**，已有磁盘缓存 + 现成 IPC `aowu:resolve-mp4-url`（`aowu/url-resolver.ts`），
  在线播放可直接复用。
- **Girigiri**：**只有 m3u8**，靠隐藏 BrowserWindow + webRequest 拦截捕获
  （`girigiri/download.ts`），cookie 会话强依赖。Chromium 原生不支持 HLS 播放，
  主进程已有 `parseM3u8()` 但那是下载管道不是播放能力——在线播放要引播放层
  HLS 库或走别的路，动手时决策。
- **渲染进程现状零播放器**：本地库播放走 `shell.openPath` 外部播放器；
  `WatchHere.tsx` 的源 chips 全部 `window.open` 外跳。`<video>` 标签、hls.js
  等依赖一概没有，全新引入。
- **自定义源数据结构已存在**：`animeTrackStore.ts` 的 `AnimeBinding`
  `{ source: 'Xifan'|'Girigiri'|'Aowu'|'Bilibili'|'Custom', sourceTitle,
  sourceKey, sourceUrl? }`，编辑走 MyAnime 的 EditBindingsModal，无需动存储层。
- **B 站嵌入（2026-07-06 已验证，别重测）**：官方外链播放器
  （player.bilibili.com/player.html，参数 bvid= 或番剧 ep_id=）**不被
  X-Frame-Options 拦**，跨域 iframe 能加载；Electron 临时窗口实测 UGC 和
  番剧都能匿名解码播放（360p/512×384 画质上限），**但 ~16s 后自动暂停并盖
  「下载客户端/进入哔哩哔哩观看」引流层 = 匿名内嵌只等于试看**。完整观看
  需要 B 站登录态。**用户否掉试看定位（2026-07-07）**：要像 BGM 那样登录拿
  cookie 完整观看。定下的方案：BGM 式登录窗（`session.fromPartition('persist:bili')`，
  用户自己在窗里登录）+ 播放页用 **`<webview partition="persist:bili">`** 嵌
  官方外链播放器——webview 里 B 站页面是顶层文档，cookie 是第一方，绕开
  iframe 第三方 cookie 分区问题；需在 BrowserWindow webPreferences 开
  `webviewTag: true`。登录后能否完全解除试看待真实账号实测（大会员专享内容
  仍受账号权益限制）。**播放页不放「浏览器打开」按钮**（用户定调 2026-07-07，
  交互稿已移除，实装时同样不放——外跳入口只在追番卡片的源 chips 上）。
  **API 自研播放器路线已评估并否掉**：B 站无面向第三方的官方播放 API；社区
  文档化的 web playurl 接口需要 cookie + WBI 签名，正片为 DASH 音视频分轨
  （得引 dash 播放层）+ CDN Referer 注入，工程量大且签名算法常变，不值得。
- 附带教训：在用户 Chrome 的自动化会话里 B 站主站和外链**都**卡在加载动画
  （无报错、playurl 正常返回也不播，环境问题），用浏览器验证 B 站播放不可靠，
  要验证就直接起临时 Electron 窗口测。
- renderer 无 CSP meta、无 onHeadersReceived 处理——引入远程媒体/iframe 时
  是空白地带，注意别顺手放开不必要的口子。

### 稀饭在线播放实测与 origin 根治（2026-07-08 / 09）
**现象**：稀饭源在线播放报「视频加载失败 —— 播放地址可能已过期」，但**同一条直链
浏览器 / 打包版都能播**。用户明确「浏览器能访问我应用就能访问才对」，一度怀疑登录
失效 / 缓存问题。

**走过的弯路（三次误判，都实测排除了，别重来）**：
- ❌ **HEVC 解不了** —— 实测 `canPlayType('…hev1…')` 返回 `probably`，裸 `<video>`
  直接能播该文件（1920×1080 canplay），排除。
- ❌ **Chromium 缓存中毒** —— 让用户清空 `~/Library/Application Support/mapletools/Cache/`
  （185MB）后重播，**仍失败**，排除。
- ❌ **没登录 / 会话失效** —— 稀饭主线是匿名可取的 pan.wo.cn 签名直链，与登录无关。
- ✅ **真因 = 页面 origin**：对比复现——`http://localhost`（dev）origin 下，主线-1
  （apn.moedot.net→302→pan.wo.cn，响应带 `content-disposition: attachment`）报
  **code 4**，主线-2（play.xfvod.pro 直链，无该头）能播；`file://`（打包）origin 下
  两条都能播。差异就是 **带 `content-disposition: attachment` 的跨源媒体被 Chromium
  媒体元素策略拦下**。所以只在 `npm run dev`（localhost）复现，打包用户不受影响。

**根治（dev / 正式都不受 origin 限制，`src/main/shared/media-proxy.ts`）**：
- 新自定义协议 `mtmedia://`（`registerSchemesAsPrivileged` 标 standard/secure/stream/
  supportFetchAPI/bypassCSP）。主进程 `net.fetch` 取远端流，**只回传 content-type /
  content-range / accept-ranges / content-length，剥掉 content-disposition**，同源回给
  `<video>`。渲染层 `toMediaProxy()` 把 http(s) 直链包成 `mtmedia://media/?u=…` 再喂 src。
- 用 `net.fetch` 而非 `netRequest`：后者 `Buffer.concat` 全量缓冲，几百 MB 视频爆内存；
  net.fetch 返回可读流边下边播，且同样走 Chromium 网络栈 + 系统代理（合红线）。
- **不缓存签名直链、每个 Range 重走 302 取新鲜链**是故意的：pan.wo.cn 单条签名链有
  并发限制，复用一条打多个并发 Range 会 `readyState=0` 卡死（曾试缓存优化直接把 ep1
  卡死，已回退）。
- **切集一直卡加载的根因**：`protocol.handle` 的 `request.signal` 在渲染进程取消时
  **不触发**（实测），旧集的流没被中断、在后台续下占满 pan.wo.cn 并发 → 新集一直转圈。
  修法：自建 `AbortController`，返回流的 `cancel()` 里 `ac.abort()` + `reader.cancel()`。
  修完切集实测 currentTime 实时推进、readyState=4 稳定；吞吐 8.3 MB/s（curl 直取 9.7），
  够 ~3Mbps 视频。
- 附带排除：控制台「sysctlbyname for kern.hv_vmm_present failed」是 macOS 上 Electron
  每次启动都打的无关噪声，与卡加载无关。

**线路兜底（用户定调 2026-07-09）**：`<video>` onerror 时——稀饭模板集先回源解析真链
重试**同线路**一次；仍失败则标记本线路已试（`triedLinesRef` Set），切到「含第 N 集且
未试」的下一条线路的**同一集**（用户强调：不是跳到第二集）；三条都试完才报「所有线路
都播放失败，可切网站或稍后重试」。`triedLinesRef` 在换集 / 手动切线路 / 点重试时清空。
（原理图见 DEVLOG「在线观看」条目下三张 SVG。）

**结论**：稀饭源应用内在线播放（dev + 打包）实测通过并已 push。Aowu 复用同一 mtmedia
代理，待站点恢复实测；Girigiri（HLS）仍待 hls.js。

### Girigiri 站点换域名 + netRequest 重定向 bug（2026-07-10）
**现象**：用户在应用里用 Girigiri 搜「我爱你」→ `Redirect was cancelled`；播放页里的
懒式搜索同样报这个。同一关键词在浏览器里搜得到。

**根因**（两层，缺一不可）：
1. 站点主域从 `bgm.girigirilove.com` 换到 **`ani.girigirilove.com`**，旧域名 301
   （`curl -I` 实测 `HTTP/2 301` + `location: https://ani.girigirilove.com/...`）。
2. Electron `net` 在 `redirect:'manual'` 下**不把 3xx 当 `response` 抛出** —— 它发
   `redirect` 事件并挂起请求，不调 `followRedirect()` 就把请求作废、从 `error` 抛
   `Redirect was cancelled`。而 `HttpSession.get()` 恰恰依赖「manual 拿 3xx → 读
   `Location` → 逐跳 ingest `Set-Cookie` → 跟下一跳」。这条路在从 Node-`https`
   迁到 `netRequest` 时就断了，只是站点一直没 3xx，**潜伏到换域名才炸**。

**修法**：`netRequest` 在 manual 模式下接 `redirect` 事件，把 3xx 的 status/headers
原样 resolve 出去；`BASE_DOMAIN` 改新域名并 export，`download.ts` 注 cookie 也跟着它走
（写死旧域名的话，cookie 落在错误 domain 上，注了等于没注）。

**教训**：`redirect:'manual'` 不是「返回 3xx 原始响应」那么简单，Electron 要求你显式
接 `redirect` 事件。任何用 manual 的地方都要检查有没有接。别再把「HttpSession 能逐跳跟
重定向」当成既成事实——它直到今天才第一次真的跑通。

**已验**（CDP 驱动真实 app）：`search('我爱你')` → 返回 `needs_captcha`（正常验证码流程，
不再是重定向错）；`resolveEpUrl` 传**旧域名** `bgm.girigirilove.com/playGV27030-1-1/`
也能跟过 301 拿到 m3u8。新域名解析 1.3s / 0.5s，播放回归正常（1920×1080、零 error）。
脚本 `verify-giri-domain.mjs`。

### Girigiri HLS 落地纪要（2026-07-10）
**已经做过的事**（避免重新踩坑）：
- **地址不用截流，HTML 里就有**（2026-07-10 实测推翻了初版设计）：播放页服务端返回的
  HTML 里直接带 `player_aaaa`（MacCMS 通用结构，与稀饭同源），`encrypt=2` 即
  base64 再 urldecode 就得到播放地址，`curl` 就能拿到、不需要执行 JS。所以
  `girigiri:resolve-ep-url` 首选 `resolveEpPlayUrl()` 解析（一次 GET，实测 0.7s，
  冷启动 2.5s），**截流（`captureM3u8`）降级为兜底**，只在站点改版解析不到时才走。
- **地址没有时效，不需要缓存**（用户提出「是不是像嗷呜那样 24h 签名链、能不能缓存」，
  实测否）：girigiri 的地址是**纯静态路径**，无 `auth_key` / `expires` / `token` /
  `sign` 任何查询串；分片只带 UA 就能匿名取（`0000.ts` → 200 / 1.2MB）。既然没时效，
  也就不存在「缓存能省掉重新截流」的收益 —— 解析本身就只是一次几百毫秒的 GET，
  加缓存反而多一层失效风险。**别再回到「缓存 m3u8 地址」这个方案**。
- **Girigiri 不全是 m3u8**：部分老番线路直接给 `.mp4`
  （`ana.girigirilove.top/zijian/oldanime/2024/0607/GrendizerU/01.mp4`）。这暴露了截流路
  的一个**真 bug** —— `captureM3u8` 只拦 `*.m3u8`，碰到 mp4 线路会白等到超时再失败。
  改成解析后，渲染层按后缀分流：`.m3u8` 交给 hls.js，`.mp4` 走和稀饭一样的直喂路径。
  **别假设 girigiri == HLS。**
- **播放层用 hls.js**（Chromium 不原生支持 m3u8）。`AI_GUIDELINES` 技术栈清单里没禁
  HLS 库，且 CLAUDE.md / 本文档早写明「在线播放要引播放层 HLS 库」，不是临时起意。
- **必须重写播放列表**：hls.js 是在**渲染进程里**用 loader 逐条取变体列表 / 分片 /
  `#EXT-X-KEY` 密钥的，拿原始 CDN 地址会被跨源策略拦。所以 `media-proxy.ts` 检测到
  m3u8 就整份读出来，把每条 URI（含 `#EXT-X-KEY` / `#EXT-X-MAP` 的 `URI="..."` 属性）
  重写成 `mtmedia://`，相对地址按**重定向后的最终列表地址**解析。列表不透传 Range、
  不透传 `content-length`（重写后长度变了），并保留 `#EXTM3U` 开头校验（取到 404 页 /
  反爬 HTML 时原样回，别把 HTML 逐行当分片重写）。
- **两个「想当然」被实测打脸，别再重来**：
  1. 以为要给 `mtmedia` 加 `corsEnabled: true` + 回 ACAO 头，否则 hls.js 的 fetch 会被
     CORS 拦 → **反例实测：两者都不加，fetch / XHR / 播放全部正常**。`corsEnabled` 是
     主动 opt-in 进 CORS 检查，加了才需要 ACAO 去满足它。已回退，`index.ts` 留反向注释。
  2. 以为 hls.js 默认 `XhrLoader` 对自定义协议没保证、得换 `FetchLoader` → **实测
     `mtmedia://` 上 XHR 与 fetch 都直通，两种 loader 都能播**。已回退成默认 loader。
- **兜底接线**：hls.js 的 `fatal` 错误接到现有 `tryNextLine()`（换下一条线路的同一集）；
  非 fatal 交给 hls.js 自己重试。HLS 时 `<video>` 不设 `src`、不挂 `onError`，免得同一次
  失败触发两遍换线路。
- **验证手法（可照抄）**：把 `out/renderer` 用本地 http 托管 + 喂 `ELECTRON_RENDERER_URL`，
  页面 origin 就是 `http://127.0.0.1`（与 `npm run dev` 同类 origin），再用 CDP 驱动真实
  app 跑断言，脚本在会话 scratchpad（`verify-hls.mjs` / `verify-hls-key.mjs`）。已验：
  公开 HLS 测试流 5 变体 / 64 分片全部重写、分片实取 1.9MB、hls.js 零 error、
  readyState=4、1280×720 时间轴推进；合成 AES-128 列表的密钥 / MAP / 嵌套相对 / 绝对
  分片地址均正确重写，`METHOD` / `IV` 无损。**girigiri 真站点端到端已验**：直接打
  播放页（不经搜索，故不碰验证码）→ 解析出 `…/ReZeroS4CHT/01/playlist.m3u8` → 经
  mtmedia + hls.js 真播起来（1920×1080、duration 1420s、readyState=4、时间轴推进、
  零 error）；mp4 线路解析出 `…/GrendizerU/01.mp4`。脚本 `verify-giri.mjs`。
  实测的这几部番列表里**没有** `EXT-X-KEY`（不加密），但下载器有解密分支说明有内容会
  加密，所以重写仍覆盖 key URI。

### B 站扫码登录（2026-07-10，别再回到 web 扫码）
**已经做过的事**（避免重新踩坑）：
- 一版是弹 BrowserWindow 加载 `passport.bilibili.com/login` 让用户扫官方页面的二维码。
  **B 站 2026-06 起收紧 web 端扫码接口（`/x/passport-login/web/qrcode/*`）风控**：手机
  确认那步直接弹「API校验密匙错误」，登不进去（downkyi 等工具同期一起中招）。这不是
  UA / buvid / bili_ticket 缺失能救的，别去补设备指纹 cookie。
- **改用 TV 端接口** `/x/passport-tv-login/qrcode/{auth_code,poll}`：appkey + 固定 appsec
  做 md5 签名，不吃 web 那套风控。参考实现是同作者的 biu 项目（`src/service/passport-tv-login-qrcode.ts`）。
- TV 端登录**不发 `Set-Cookie`**，cookie（SESSDATA / bili_jct / DedeUserID…）直接在 poll
  的响应体 `data.cookie_info.cookies` 里，要自己逐条 `session.cookies.set()` 进 `persist:bili`。
- 二维码在**主进程**用 `qrcode` 包生成 PNG data URL（白边烤进图里，深色主题不反色），
  渲染层只 `<img src>`——渲染层不引新前端库。
- 轮询每 2s 一次，**不算违反「不自动重试」红线**：这是 B 站定义的扫码协议，只在弹窗
  开着且二维码有效时问，关窗即停；请求真出错就停下报错。
- 登录入口在「设置 → 通用 → B 站账号」和播放页提示条，共用 `components/BiliLoginModal.tsx`。

### B 站自研播放（2026-07-10，推翻了「API 自研播放器路线已否掉」）
**已经做过的事**（避免重新讨论）：
- **外链播放器 `player.bilibili.com/player.html` 是死路**，四条毛病全是它写死的、登录也解不开：
  画质菜单里列着 1080P 但**选了没反应**（实际锁 360P，那句「进入哔哩哔哩，观看更高清」
  就是它在明示）、暂停弹推荐位、盖引流层、合集拿不到分 P 列表。**别再退回去。**
- **当初否掉自研的理由死了两条**：「WBI 签名常变」——TV appkey 的 appsec 是固定常量，
  md5 排序拼接即可，压根不碰 WBI；「CDN Referer 注入」——`mtmedia` 代理加个头就行。
  只剩「DASH 分轨」是真实成本，引 `shaka-player` 解决。
- **1080P 只在 DASH 里**（实测）：mp4 直链（`fnval=0/1`）的 `accept_quality` 只有 `[64,16]`，
  容器封顶 720P。想要 1080P 必须音视频分轨 + MSE，没有第二条路。**别再试图找 1080P 的
  mp4 单文件。**
- 画质由**登录态**决定，不是参数说了算：匿名最高 480P，登录后 `dash.video` 里才出现
  `id=80`；1080P 高码率（112）还要大会员。所以 playurl 必须带 `persist:bili` 的 cookie
  （`netRequest` 的 `session` 选项）。
- **MPD 里只放 avc1**：B 站每档画质同时给 avc1 / hev1 / av01，编码不同不能塞进同一个
  AdaptationSet；avc1 是唯一各平台都能硬解的。
- **画质档只列真有的**：`accept_quality`（账号能选）∩ `dash.video`（稿件实际有）。
  否则就复刻了外链播放器「摆着点不动」的坑。
- **ABR 关掉**：用户点 1080P 就得一直是 1080P。切档走 `selectVariantTrack`，不重新 load。
- **`mtmedia` 依旧不加 `corsEnabled`**：shaka 的 fetch 和 hls.js 一样直通（011 前面已实测过
  一次，这里复验通过）。加了反而要补 ACAO。
- **集数 = 分 P**：`x/web-interface/view?bvid=` 匿名可取，`pages[]` 里 `page` 就是链接上的
  `&p=N`，`cid` 拿去换播放地址。单 P 稿件 pages 长度为 1。
- **番剧（ep 链接）没动**：走的是另一套 pgc playurl，仍嵌外链播放器，画质照样被锁。
- 验证手法（可照抄）：CDP 驱动真实 app，`file://`（打包）与 `http://127.0.0.1`（等价
  `npm run dev` 的 origin）各跑一遍——011 早期栽在 origin 上，媒体类改动两种都要验。
  脚本 `verify-bili-play.mjs` / `verify-bili-dev-origin.mjs` / `verify-bili-qr.mjs`。
