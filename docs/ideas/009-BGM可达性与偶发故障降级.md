> 开始：2026-06-02

## 🎯 这个阶段的产物

搞清 bgm.tv **非限流**故障（偶发 502 / 超时）的根因，并据此给搜索/详情一条**对症的
降级线路**，让 BGM 在这些故障下仍可用 —— 用户不再「停机半天后第一次搜索就超时、
又没别的办法」。错误提示**如实**反映行为（不谎称已自动重试）、**对症**（偶发故障 /
限流 / 站点不可达分清楚）。

用户视角：bgm.tv 抽风时，搜索仍能出结果或给出明确可行的下一步，而不是干超时。

## 📖 详细说明

### 502 / 超时根因诊断

**目标效果**：明确 bgm.tv 偶发 502/超时属于哪种 —— Cloudflare 盾挑战 / 纯 CDN 网关
抖动，作为选降级方案的依据（不同根因对症的方案不同）。

**复现**：停机 12h+ 后第一次搜索（Anime Info 顶部搜索框，如「天才千金的魔法革命」，
类目「漫画小说」）→ 首次「请求超时」（10s 无响应）；点 Try again 后变「HTTP 502」。
报错原文字串：`BGM 返回 HTTP 502`。诊断埋点上线后，点错误面板「Show details」可见
`status=.. server=.. cf-ray=.. cf-mitigated=.. cf-cache-status=.. via=.. content-type=.. | body[0:300]=..`；
`npm run dev` 终端同时打 `[bgm-search-diag] ...`。失败是偶发的，可能要多搜几次才逮到。

**已经做过的事**（避免重新踩坑）：
- **已排除「是我们限流」**：BGM 搜索是**纯 HTML 抓取**（`bgm.tv/subject_search`，无 API
  主路，见 `search.ts:80` 注释）；新会话 `consecutiveSearchTimeouts=0`、搜索路径**无**
  持久化熔断（008 的熔断只在 api.bgm.tv 那条）；12h 空闲滚动窗口早衰减干净。
  → 这是纯**上游 / CDN 事件**，不是我们这侧。
- 502 = CDN 边缘节点没从源站取到正常响应；浏览器能开是命中了别的健康节点 / 热连接 /
  时间差 —— 「浏览器行、app 这次拿 502」不矛盾。
- `net`（Electron net）已经走 Chromium 网络栈（系统代理 / PAC），但**缺**真浏览器才有的
  TLS 指纹 / HTTP2 / cf-clearance（过盾）/ 热连接 —— 这是「浏览器稳过、裸 net 稳挂」
  的可能原因。

### 降级线路选型（等诊断结果再定，先记已决策 / 否过的）

**目标效果**：非限流故障下搜索/详情仍可用，**尽量不牺牲**现有的中文别名 / 系列召回。

**已经做过的事 / 已决策 / 否过的方向**：
- **详情页已经做了**：任何 api 失败 → `bgm.tv` HTML 兜底（`detail.ts:225`，008 第二步，
  已从「仅 RateLimitError」放宽到 502/超时/网络抖都兜底）。**搜索本来就是 HTML、没有任何
  兜底** —— 这才是用户实际撞到的痛点。
- **候选 A — 隐藏 BrowserWindow「像真浏览器那样」加载 bgm.tv**（用户提的方向；Girigiri
  下载已用此模式抓 m3u8，项目有先例）。优点：仍是 bgm.tv HTML → **别名/系列召回零降级**；
  真浏览器能过盾 / 命中健康节点。缺点：重（内存/启动）只能当**兜底**不能当主路；对**纯随机**
  CDN 抖动治标有限。**红线细化**：仍须保留限流页检测 + 节奏，**不拿它去绕限流**（用户已知情
  同意这个区分）。
- **候选 B — api.bgm.tv 搜索兜底**。轻、简单，但用 BGM **自带匹配** → **无中文别名/系列召回**，
  俗称/系列名搜会降级（不如主线精准）。
- **粘滞**：检测到 bgm.tv 异常后约 **10 分钟**内搜索直接走兜底，不再每次先撞 10s 超时；
  窗口过了下次搜索**自然**再试主线（不主动探活，合手册「零主动探测」红线）。窗口长度用户已选 10min。
