# 012 - 网页版

> 开始：2026-07-13

## 🎯 这个阶段的产物

把 MapleTools **的一部分功能**搬上网页，让没装桌面应用的人（尤其 Android / iOS
移动端）在浏览器里也能用。**不照搬全部功能** —— 涉及本机的那些（本地库扫描 /
文件浏览器 / ffmpeg 缩略图 / Windows 回收站 / 邮件任务 / WebDAV 本地同步等）
网页版天生不适用，一律不做。

先做**暂时不依赖本机**的功能，三个核心：

1. **番剧周期表**（BGM 日历，只读抓取 + 缓存，最轻）
2. **我的追番列表**（用户数据，要存储、后续要登录归属）
3. **在线观看**（最重 —— 抓流 + 流代理都得搬服务端，带宽/成本是硬约束）

网页版与桌面 exe 版默认仍是**两拨独立用户、各持一份独立数据**：应用用户的追番存在
本机，网页用户的追番存在服务端、按账号隔离。2026-07-24 起 app 可选登录网页版账号，
手动上传 / 拉取追番；不登录时仍完全独立、离线可用。

## 🧭 用户已定的调（别再回头讨论）

- **功能是子集**：本机相关功能不做网页版；先上不依赖本机的三个核心。
- **数据独立**：网页版数据服务端单独持有，和 exe 版完全分开，两类用户各走各的。
- **注册登录优先级仅次于网页版本身**：先把网页版跑起来（可以先不带账号，
  或先用最简单的方式占位），登录 / 每用户独立数据紧随其后，不是第一步。
- **登录后每个用户数据独立**：注册登录是为了「这个人的追番只属于这个人」。
- **规模小但开放**（2026-07-15 更正）：用户量小（GitHub 目前 6 个 star ≈ **至少** 6 个潜在
  用户），但**不封顶、开放注册** —— star 的人互不认识，发不了邀请码。早先「封顶 6 人 / 邀请制」
  的假设作废。不为规模做重预留（不上负载均衡 / 分布式 / 缓存集群），但账号体系要让陌生人自助注册。
- **成本红线**：先用**免费服务器，或极低成本 —— 一个月几块 / 一年不超过 50 块**，
  但**不要太垃圾**。这是硬约束，直接决定选型（见「待调研」第 1 条）。阿里云那台
  内存 / 空间都不够，本就要换。
- **同仓库、结构隔离、绝不影响应用**：网页版**必须在同一个仓库**（不接受另开 git
  项目），但**绝不能动到 / 影响应用现有功能**。已定：放顶层 **`web/` 子目录 + 自带
  独立 package.json / node_modules**，物理隔离（见「详细说明 · 网页项目放哪」）。功能和
  效果**参考**应用版（对齐 UI / 交互），而不是去复用会跟 Electron / IPC 纠缠的代码。
- **dev 天然分两版**：既然是独立项目，「应用版 / 网页版分开跑」自动满足 ——
  各自独立 dev、互不干扰（比一个工程里塞两种运行模式更干净）。
- **移动端顺带解决**：网页版本身就是 Android / iOS 的答案，不再单独做原生 App。

## 📖 详细说明

### 网页版到底做哪些功能（2026-07-15 产品思考，功能线未完全拍板）

用户重新审视网页版范围，核心几条结论：

- **网站的真正目的 = 快捷定位**：省去「每次去 xifan 搜索 → 找到 → 看」这一套。
- **「搜索 vs 直接访问 xifan」的悖论解法**：快捷定位**不是做个更好的搜索框**，而是**对已在追的番免搜索**。
  拆成两种意图 —— ① **首次发现**（找新番看）才需要搜索，把 xifan/girigiri/aowu/自定义源聚合成一次搜
  是小赢，但跟直接逛 xifan 差别不大；② **日常追更**（我在追的那几部更新没?下一集在哪看?）**才是快捷
  定位的主场**：打开网站 → 追番列表 / 周历表高亮今天更新 → 一键跳下一集续播，**全程不搜索**。这正是
  app 版 011 在线观看 + bindings 做的事。
- **推论：在线观看要有价值，离不开追番这层**。没有追番，在线观看退化成「又一个搜索站」（= 用户担心的
  「和直接访问 xifan 没区别」）；有了追番，搜索只是配角（首次加番用一次，之后靠列表一键续播）。所以
  **追番是在线观看的「快捷定位引擎」**，不是可有可无。
- **坚果云数据分裂**：追番 / 锦囊妙计 / 妙语库 的 app 版数据在**作者私人坚果云**里。公网 web 用户
  （≥6 个 star 者）**不能也不该**共享作者的坚果云 → 每个 web 用户必须有自己的服务端数据，**这个「分裂」
  是多用户隔离的必然，不是 bug**。唯一别扭的是作者自己会有「app 追番（坚果云）+ web 追番（服务端）」两份
  → 要么接受（作者继续用桌面版），要么以后做 web 账号 ↔ 坚果云 桥接（后置）。
- **建议功能线**（待用户最终拍板）：
  - **web 核心 = 周历表 + 在线观看，追番作为在线观看的快捷定位引擎**。web 版追番可比 app 版更瘦，职责只是
    「记住在追什么 + 每部的源 / 进度 → 一键续播」，不搬 app 那套 PDF 衍生的完整追番系统。
  - **锦囊妙计 / 妙语库 保持 app-only**：个人学习 / 参考数据、多用户价值低、跟坚果云（妙语库还带图片同步）
    深度纠缠，搬 web 得不偿失。
  - 所有本机功能（本地库 / 文件浏览器 / 回收站 / 邮件 / WebDAV）天然不做 web。

### 追番联动 & 不丢数据（2026-07-15 已定原则，稍后实现）

「app 老用户想同时用 web」和「web 用户又装了 app」两种都要覆盖，核心是**别让瘦的 web 数据冲掉富的
app 数据**。

- **web 账号 = 同步枢纽**：app 可选「登录 web 账号同步」，追番对着服务器 API 双向同步（app↔app 设备间
  仍走坚果云，两条 merge 都是字段级、可共存）。无论从哪端起步都统一到 web 账号：
  - **先 web 后 app**：app 里登录 web 账号 → 拉下 web 追番（瘦记录进 app，normalize 补默认）→ 之后双向。
  - **先 app 后 web**：建 web 账号 → app 推送富记录（服务器整条留存，见下）→ web 显示瘦视图。
- **铁律：同步是「字段级 merge（patch）」，绝不「整条替换」**。web 只写它拥有的字段（status / episode /
  favorite / tags），对 app-only 字段（goodEpisodes / aliases / bindings / novel…）**保持沉默；沉默 ≠ 置空**
  → app 的富字段永不被 web 碰掉。用户担心的「被少数据覆盖」只会发生在幼稚的整条覆盖，我们不这么写。
- **服务器 schema 容忍**：tracks 表 = 索引用的瘦列（status / episode / favorite… 供 web 查询 / 展示）+ 一个
  `extra` JSON 列存 app-only 字段 → app 的富数据能原样过服务器往返，不丢。
- **同字段冲突**：按字段（或记录）`updatedAt`「后写者胜」（≤ 几十人，够用，不上 CRDT）。
- **新番只在一端**：只在 web → app 收到当新 track、normalize 补默认（不是丢，是本来就新）；只在 app → web
  显示瘦投影、app 富字段原地不动。

### 追番同步 · 落地决策（2026-07-23，确认走上面那条，WebDAV 降级）

上面 07-15 的原则不变（web 账号当枢纽 / 字段级 patch / `extra` 存 app-only 字段 / `updatedAt` 后写者胜）。
这次动手前把剩下的口子补上，**有一条是当时没想到的漏洞**：

**① 形态：网页实时直连，app 手动覆盖式拉取 / 上传**（用户 07-23 定）。

```
网页  ──每次操作直接写──▶  服务器（唯一真相源）
                            ▲ │
              app 手动「上传」│ │「拉取」  ← 整包覆盖，方向由用户声明
                            │ ▼
                        app 本地 localStorage
```

网页版本来就没有「同步」概念（每次改动直接打接口，无需改动）；同步的负担全在 app，就两个按钮，UI 复用现成的 `AnimeSyncBar`。

**② 因此不需要「删除墓碑」。** 曾一度打算做（软删 + 墓碑下发），后来砍了 —— 墓碑是**逐条 merge** 才需要的：merge 模型下「服务器少一条」跟「本地新加还没传」长得一样，删除会复活。覆盖模型里方向是用户声明的，整份替换，删了就是没了。**别再绕回来加墓碑**，除非哪天真要做双向自动 merge。

**③ 代价（认下来的）：两端各改各的时候，后点的方向会盖掉另一边。** 中午在网页 +1 集，晚上在 app 改了别的直接点「上传」→ 中午那次没了。跟现在 WebDAV 的坑是同一个，不是新增痛点。

护栏用 **`tracks_rev`（每用户一个递增号，服务端每次写入都 +1）**，不是比时间戳 —— 时间戳要靠两端时钟，设备时间不准就会判错。app 记住上次同步时的 rev，上传时带上：

```
app 带的 baseRev == 服务器当前 rev  → 直接覆盖
          不相等                    → 409 + 提示「服务器上有你没见过的改动」→ 用户选「先拉取」或「强制覆盖」
```

**④ 富字段不会被瘦记录冲掉**（一度以为是选型理由，其实不是）：服务器存完整记录，网页端的写入本来就是字段级 patch、根本不碰 `extra` → app 的 `bindings / notes / goodEpisodes` 原样往返。两种模型下都安全。

**⑤ app 不登录也能用**（用户 07-23 定）：追番仍先写本地 `localStorage`，登录后才出现拉取 / 上传。老用户升级无感、断网可用。代价是「本地态」和「已登录态」两条路径都要维护 —— 有意付的成本。

**⑤ WebDAV 从「同步通道」降为「定期备份」**（改动 07-15 的说法：那时 app↔app 仍走坚果云）。理由：真相源只该有一个，两条 merge 通道并存迟早互相打架。配了 WebDAV 就由 app 定期自动推快照（用户无感），原来的手动推送/拉取按钮保留做应急。**不做服务器端直连 WebDAV** —— 那要求服务器保存用户的坚果云账号密码，多一份高价值凭据在 VPS 上，不值。

**现成的部分**（省了不少事）：`tracks` 表已有 `updated_at` + `extra TEXT '{}'`（目前恒空、没启用）+ 写入本来就是字段级 patch；app 的 `AnimeTrack` 也早有 `updatedAt`（每次 upsert 自动 bump）。手动条目的负数 `bgmId` 在 `UNIQUE(user_id, bgm_id)` 下照收。

**实施顺序**（每步都能单独验、单独上线）：

```
1. [x] 服务端：extra 真正读写 + tracks_rev + 批量拉/推接口
2. [x] app：账号登录（JWT 存主进程，preload 只回登录状态）
3. [x] app：两个按钮接服务器 + 上传前的 rev 冲突提示（2026-07-24 完成）
4. [ ] WebDAV 改定期备份（当前手动按钮继续保留做应急，且与网页版 rev 完全独立）
```

**部署避坑：同步接口与桌面端必须配套上线。** 服务器还停在旧版本时，
`GET /api/tracks/sync` 会被 SPA 兜底吞掉，返回 `200 + index.html`；同一路径的
`POST` 则返回 404。桌面端不能只凭 GET 状态码判断接口存在，必须同时校验
`Content-Type: application/json`；遇到 HTML / 404 应明确提示「网页版服务尚未更新」，
不要归类成普通上传或网络失败。部署后要分别验证一次 GET 和 POST，不能只测健康检查。

**数据补全避坑：客户端的“未知”不能覆盖服务器已知值。** 老 app 没持久化
`airWeekday`，上传时的 `0` 只表示“不知道”，不能把周历已有星期清空。新 app 在
`AnimeTrack` 持久化 1～7；服务器对本季旧记录从周历补星期、首播日和缺失封面，
但这种系统元数据补全不递增 `tracks_rev`，否则用户只是打开页面也会制造同步冲突。

**顺序避坑：API 返回顺序不是 app 添加顺序。** 服务端按 `updated_at DESC` 返回，
而 app 的 `Map` 保存原始插入顺序、页面再反转显示。上传时必须在 `extra.appOrder`
显式携带插入位置；拉取旧数据没有该字段时，用既有 `startedAt` 恢复，不能直接采用
服务端数组顺序。

**封面避坑：路径式代理只接受 `lain.bgm.tv`。** 手动条目可能使用任意 HTTPS 图床；
只有 lain 地址才能剥掉 host 改写成 `/api/cover/pic/...`。把外站 `/files`、
`/resource/news` 等路径也塞进 BGM 代理，会稳定得到 403/404 和裂图。

### 最大的坎：抓取 / 流代理必须从「主进程」搬到「真服务端」

现在的架构里，渲染进程**只**通过 preload IPC 找主进程，所有网络 / 抓取都在
主进程用 Electron `net` 走系统代理（`netRequest()`，见 CLAUDE.md 红线）。网页版
没有这个主进程：

- **浏览器抓不了那些站** —— 跨域被 CORS 挡、没有系统代理会话、验证码 / cookie
  会话那套也做不了。所以 bgm / xifan / girigiri / aowu / bili 的抓取逻辑得整体
  搬到**真服务端**，前端改成走 HTTP(S) 调后端（而不是 IPC）。
- **在线观看更重** —— 应用里在线播放靠 `mtmedia://` 自定义协议在主进程代理流 +
  剥 `content-disposition` 头（见 011）。浏览器没有自定义协议，等价物是**服务端
  流代理端点**：视频要过服务器一道 → **带宽 / 成本是硬约束**，叠加「阿里云资源
  不够」这条，很可能直接决定在线观看在网页版可不可行、或要不要降级方案。
  **这条要最先想清楚**，别等三个功能都做完才发现流量扛不住。

> 一个自然的分期就是按「服务端负担」从轻到重排：周期表（只读抓取+缓存）→
> 追番（用户数据存储）→ 在线观看（流代理，最重）。先用周期表把

**视频这块的「三选二」（认清，别对免费机抱不切实际期望）**：①服务器便宜 · ②播放流畅
· ③用户不用自己挂代理 —— 最多同时满足两个。「用户不用代理」= 视频得从你服务器中转

**这只影响在线观看**，周期表 / 追番不受此约束。到在线观看那档按源做原型：哪些能让浏览器
直连（渐进式 mp4 多半能，HLS 因 CORS 多半得过一道）、哪些必须代理。

### 抓取复用策略（2026-07-13 已定：拷贝复用 + 换传输层，绝不碰 app）

- **能复用就复用，但方式是「拷进网页项目」，不是重构应用共享代码。** `src/main/` 的
  bgm/xifan/girigiri/aowu/bili 抓取逻辑（cheerio 选择器、URL 模板、aowu 解密、m3u8
  重写这些）拷一份到网页项目，**应用那边一行都不动、Electron `net` 依赖原样保留**。
- **只换传输层**：拷来的模块里唯一要改的是 `netRequest`（Electron net）→ 后端自己的、
  **可挂代理**的 Node HTTP 客户端。**做成 `netRequest` 的 drop-in 替身**（同签名 / 同返回
  形状），拷来的解析代码近乎零改动，日后从 app 往这边搬 bug 修复也好对。
- **已知代价，认了**：拷贝 = 两份会漂移（app 修了「girigiri 换域名」那种，网页这边不会
  自动跟）。这是「绝不碰 app」硬约束的必然成本；缓解靠「拷得越原样越好、diff 易移植」，
  **先不抽公共包**（那会反向耦合 app，违背独立项目初衷，YAGNI）。
- **代理怎么挂（分路线）**：走 A（serverless）**没有「机器」可装 Clash，出站就是从边缘节点
  直接 `fetch`** —— 周期表 / 追番的目标（BGM 等）边缘直连即可、不用代理。「按域名挂代理分流」
  是走 B（Oracle VPS）装 Clash 那套，和应用「跟系统代理走」一致。嗷呜这类屏蔽境外节点的国内
  专供源、以及视频流代理，无论哪条路都留到在线观看那档专门解决。

### 安全审查（2026-07-17，线上已有真用户数据后做的第一次）

**审出来的洞（都已修，见 DEVLOG 2026-07-17）**：

- 🔴 **node 绑 `0.0.0.0:3000` + 服务器没防火墙 → 公网直连能整个绕开 nginx**。`curl http://103.250.172.17:3000/api/auth/questions` 直接 200。这意味着谁都能用**明文 HTTP** 调 `/api/auth/login`（密码裸奔过公网），HSTS / 证书 / 强跳全成摆设，`X-Forwarded-For` 也随便伪造。**教训：反代场景下「后端能从外面访问」不是功能，是洞**；框架默认监听地址（`serve({fetch, port})` → `0.0.0.0`）在这里就是错的默认值。
  （会话 cookie 有 `Secure`，浏览器不会把它发到 http，所以偷不到会话 —— 裸奔的是**登录时提交的密码本身**。）
- 🔴 **`/login` 完全没限流**。连打 7 次错密码全 401、从不 429。**教训：`/forgot` 加了限流是因为「密保答案熵低」这个点显眼、想得到；`/login` 是同一类入口，却因为不显眼被漏掉 —— 挑着加 = 漏**。凡是「拿凭证来猜」的入口都得挂。
- 🟡 数据库零备份（线上已有真账号）、无 HSTS、库文件 644。

**验过没问题的**：`AUTH_SECRET` 64 位且在仓库外、`DATA_DIR` 在部署目录外、cookie 三标志（`HttpOnly; Secure; SameSite=Lax`）齐、密码和密保答案都是 scrypt、`/forgot` 限流准确在第 6 次 429。

**排查方法上的坑**：**SPA 兜底让任何路径都回 200**（`app.get('*', → index.html)`）。测 `/.git/config`、`/server/auth.ts`、`/data/web.db` 全是 200 —— 一眼看去像泄露了一片，实际 body 都是那 440 字节的 `index.html`。**状态码在这个站上没有诊断价值，必须看 body**。

**审查的结论也要照着做**：审出来「node 用 root 跑」当天没动，理由是「怕动 pm2 出事」—— 结果第二天还是得动，而且是在更赶的状态下动的。识别出的风险当场消除（见 AI_GUIDELINES「工程习惯」）。

### 改用非 root 跑应用（2026-07-17）踩的坑

- **`su user -c "cmd"` 不切工作目录**，会继承调用方的 cwd。root 的 `/root` 是 700 → 目标用户进不去 → pm2 spawn 子进程时报 **`spawn /usr/bin/node EACCES`**。这个错误**指着 node 撒谎**：`su -s /bin/bash mapletools -c "node -v"` 明明跑得通，查 node 权限、ACL、lsattr 全是干净的，真凶是 cwd。**教训：EACCES 别只盯着报错里那个路径，spawn 的 cwd 也在权限检查范围内。** 已用 `/usr/local/bin/mtweb` 把 `cd` 焊死。
- **`systemctl restart pm2-<user>` 验开机自启会假失败**：守护进程还在跑 → `pm2 resurrect` 直接连上去就退出、没 fork → systemd（`Type=forking`）等不到 PID 文件 → 报 `Result: protocol`。**这不是真故障**。真开机时没有守护进程，行为不同。要验就先 `mtweb kill` 杀干净再 `systemctl start`。
- **root 的 cron 备份会悄悄夺走库文件属主**：root 打开 SQLite 时若 WAL/SHM 不存在会新建成 root 所有 → 应用（mapletools）从此写不了自己的库，且**要等到凌晨 4 点之后才爆**。备份脚本末尾必须 `chown` 回去。
- **备份目录故意保持 root-only**：应用用户读不到也删不掉。勒索软件标准操作就是先毁备份 —— 备份和被保护的东西不能在同一个权限域里。

### 视频怎么流畅（2026-07-13 结论：靠 browser-direct + 客户端边下边看，不靠服务器）

用户担心「花钱也买不到全源流畅」。探讨结论：**视频流畅主要是代码 / 架构问题，跟服务器强弱基本
无关**，所以普通 CN2 VPS 够用、**不必为视频加码**。

- **核心 = 浏览器直连源 CDN（方案 B）**：国内盗版源的 CDN 本就服务国内、很快。让**浏览器直接从源
  CDN 播**，视频字节**不经过你服务器** → 服务器强弱与视频无关，土豆机也行；服务器只干「解析出播放
  地址」这点活。
- **边下边看 = 更深的缓冲**（不是「下载更快」，同一条路速度一样）：只要平均网速 ≥ 码率，往前多灌
  就不卡。渐进式 mp4 在 `<video>` 里浏览器本就贪婪缓冲；要更狠可自己 fetch 整集。HLS 调大缓冲窗口。
- **预下载下一集**：用户一集集看，提前灌 ep2 —— **只在方案 B 下划算**（不烧服务器带宽）。
- **跳转不是问题**（如跳到第 2 集第 10 分钟）：mp4 用 HTTP Range、HLS 用分片，从任意点开始拉 ——
  拖进度条本就这么工作，预下载只是顺路往前灌。
- **P2P / 服务器缓存：≤6 人不做**。P2P 是规模技术（要「同一集很多人同时看」，6 人几乎不可能同时撞
  同一集）；方案 B 下源 CDN 已负责分发，也不需要缓存（还惹「把盗版视频存自己服务器」的麻烦）。
- **躲不掉的残余**：挡 browser-direct 的源（HLS 要 CORS、或校验 Referer 防盗链）没法直连 → 退服务器
  中转（弱机吃力）或让用户自己挂魔法。所以「**全源**一致流畅」给不了；但**能直连的源（尤其渐进式
  mp4 的国内 CDN）在便宜服务器上就流畅**。到在线观看那档**一个源一个源实测**。

### 在线观看原型纪要（2026-07-21，稀饭已跑通浏览器直连）

到「在线观看」这一档先拿稀饭做原型，结论都是实测出来的（别再回头讨论）：

- **绕开了「服务端流代理」那条重路**（012 原本担心带宽是硬约束）：稀饭的 mp4 线（xfvod）和 HLS 线
  （playxf）**都挂在 Cloudflare 后、回 ACAO** → 浏览器能**直连源 CDN**（mp4 走 `<video>` no-cors；HLS 走
  hls.js），**视频字节不经服务器 = 零视频带宽**。app 那边「盗版 CDN 不给跨源、HLS 必须服务器代理」的假设
  **在稀饭不成立**。「视频三选二」里最贵的分支（服务器中转）这就拿掉了。
- **懒加载选线（2026-07-21 改，掉头去掉了「并行解析 + 自动选最优」）**：打开只抓 **1 次**
  （`getPlaylist`：source 1 页 → 线路 1 地址 + 正则扒源 tab 拿全部线路名），线路 2/3 **用户点了才
  `resolveLine` 抓那一条**。**为什么不再并行解析所有源自动挑最优**（原设计）：一串请求瞬间砸向稀饭像爬虫、
  会触发反爬 / 限流（项目抓取红线本就要求礼貌）；而且用户多半只看默认线路，预解析其余是白费。
  播放层按类型分派：`.mp4` → `<video>` 直连、`.m3u8` → hls.js，**直连播不了就套娃**（稀饭真实播放器
  `player.moedot.net/player/index.php?code=xfdm1&from=cf&url=`）。所以**不必预探 content-disposition / HLS
  空壳**，交给「播不了 → 套娃」统一兜（content-disposition 在 localhost 拦 = 011 老坑；CS 线偶尔是空壳
  `#EXTM3U` 9 字节，hls.js 报 manifestParsingError → 套娃）。
- **定位免验证码**：验证码闸门只在 search，**播放页 / 周表页都不闸**。当季番从**周表页**
  `/index.php/label/weekday.html` 直接拿 animeId（标题 → `/watch/{id}/…`），日常追更全程不碰验证码。老番才走
  search（验证码），且解析结果**跨用户共享缓存** → 只有第一个人解一次，其余人零验证码。
- **「比稀饭更顺」要校准预期**：视频流的瓶颈是**源 CDN 速度**，我方播放器和稀饭用**同一个 CDN** → 流本身
  **快不过**稀饭，只能靠**更深的缓冲**（hls.js 目标拉到 10 分钟垫子、暂停照灌）赢，而这**只在「网速 ≥ 码率」时
  兑现**。web 版真实价值是**快捷定位**（追番一键直连、省掉搜索-验证码-找番-点进去）+ **换集预下**（待做），
  **不是「流更顺」**。（原生 `<video>` 缓冲 30-60s 就停、暂停停下载，这也是 mp4 单流会卡、HLS+hls.js 更稳的原因。）
- **踩坑：缓冲读数别只挂 `timeupdate`** —— 它**暂停时不触发**，缓冲还在涨但界面数字冻住，看着像「暂停不缓冲」（排查了半天，其实 hls.js 后台一直在灌）。改用 `setInterval` 定时刷，再加个 `FRAG_LOADED` 分片计数：暂停时分片数还在 +1 = 确实在缓冲。
- **共享缓存**：已解析的那条按 `(animeId, ep, source)` 全站共享（进程内 Map，1h TTL）→ 刷新 / 换人秒回。
  **不并行、不预解析**（见上条懒加载）。地址是静态路径（xfvod/playxf 无签名、pan.wo.cn 入口 apn.moedot.net
  也稳），缓存安全。`fetchHtml` 带**单次瞬时重试**（稀饭偶发 "socket disconnected before TLS"）。
- **形态**：播放页 `/api/xifan/play-page?animeId=&ep=`（+ `/playlist` 拿名单+线路1、`/resolve?source=N` 懒解析单条、
  自托管 `/hls.js`）。**2026-07-22 已接上追番卡片「继续看」**（周表免验证码建绑定，见下「继续看 · …绑定」一节）。
- **残余**：真实流畅度是**纯速度问题**，Mac + Clash 绕海外够国内 CDN 慢、给不了真实答案，**要部署后手机
  无魔法国内实测**才算数。嗷呜屏蔽境外、香港服务器够不着（见「海外节点的两个坑」）。

### 继续看 · bgmId ↔ 稀饭 animeId 绑定（2026-07-22，接上追番卡片）

- **两套 id、无确定映射**：追番存 BGM id，稀饭播放页要稀饭自己的 animeId，唯一联系是标题；而 search 有验证码。
  突破口是稀饭「追番周表」的数据接口**不设验证码**——
  - **坑：周表页是 AJAX 渲染的**，静态 HTML 只有 7 个空 `#week-list{1..7}` 容器（一堆 `href="javascript:"`），
    番单在 `script.js` 里 `POST /index.php/ds_api/weekday`、body 是**中文星期**（`weekday=一`，一/二/…/日）拉。
    返回 `{code:1,list:[{vod_id,vod_name,url,vod_remarks}]}`——`vod_id` 就是 animeId，`vod_name` 中文名，
    `vod_remarks`（`"03|周一22:00"`）带更新到第几集。**别去 grep 静态 HTML 的 `/watch/` 链接（没有）**，要打这个 POST。
- **匹配模糊 → 候选让用户确认，绝不自动认**（跟 app「源只按显式 bindings 关联、绝不模糊匹配自动绑」同一条原则）：
  `locate.ts` 归一化（NFKC + 去标点空格 + 只留汉字/假名/字母数字）后按「相等 / 互相包含 / 二元组 Dice」打分排序，
  前端弹「选择稀饭片源」候选框，点一个才算。季度后缀（第二季/Ⅱ）、简繁、日文原名都可能对不齐——人眼确认一次最稳。
- **绑定表全局、不按用户**（`xifan_binding`，主键 `bgm_id`）：`bgm→xifan` 是客观事实、对所有人一样，一人确认全体命中。
  **不塞进 `tracks.extra`**——那列留给 app 富字段原样过路（同步铁律），web 不该碰；绑定是 web 自己的数据，另立一张表。
- **持久 + 跨季**：落库后周表换季、原番消失也照常开播（不再依赖周表）。定位**只对当季在播有效**（周表只列在播），
  往季旧番暂无解（search 有验证码），弹框里直说「本季周表没有」，不硬猜。
- **落集数 = 已看 +1**（`nextEp`）：「继续」就是往下看，周更用户打开时新集刚好这一集。真开到没更新的集，靠**集数网格**
  点回去即可——不指望默认落点永远对，给个能纠的口子。
- **前端不吃弹窗拦截**：绑过的卡片「继续看」直接是原生 `<a target=_blank>`（无异步），没绑的才是 `<button>`（点了去定位、
  弹候选，候选行本身也是 `<a>`）——全程真实用户手势开新标签，不触发 popup blocker。
- **播放页参考 app 播放器重做**（2026-07-22）：标题 + EP 徽标 / 线路卡片 / **集数网格**（从 watch 页 `.anthology-list` 正则扒
  整季集数、当前高亮、点即换集），玫瑰主色对齐 app/web 暗色主题。app 那套 `mtmedia://` 主进程流代理 web 没有，播放机制不照搬。
- **下载型链接按 URL 判死走 iframe**（修正原型纪要「不预探 content-disposition」那条）：`apn.moedot.net/d/…` 会 **302 跳
  `pan.wo.cn/openapi/download`**（联通网盘），`<video>` 喂它=浏览器**下载 400MB 附件**（还播不了）、白等几秒才切套娃、
  甚至触发下载器弹窗。这是**服务端 attachment、各端一致**（区别于 011 那种 origin 相关的 content-disposition），值得在
  `classify()` 里按 URL **直接判 iframe**、跳过 `<video>`；真·origin 相关的失败才留给播放层「直连失败→套娃」兜底。

### 追番目录数据源 · 本地 BGM 全量索引（2026-07-22，搜索加番）

**先分清两件事**：「能追什么番」（目录）和「能在哪看」（观看：xifan/girigiri/自定义链接）是独立的。目录只管搜得到、
加得进；片源有没有是观看层的事。所以目录数据源按「元数据全不全 + 中文名准不准 + 对服务器友不友好」选，不被片源牵着走。

- **为何本地离线档、不在线搜 BGM**：BGM 数据最全最准（全站在用），但在线 API 对「单服务器 IP 高频搜」不友好 ——
  一旦因搜索把 IP 搞进限流/黑名单，**周历和封面代理会一起挂**（都靠 BGM）。app 是每人各自 IP、分散，网页版一个 IP
  集中、风险高。故用 BGM 官方**离线档**（`bangumi/Archive`，每周三，419MB zip）灌本地库，搜索全打本地、零在线请求。
  - 不选 AniList/MAL/TMDB：资源也全但**中文名弱**（主认罗马音/英文），中文搜体验差。
- **档格式的坑**：档里 `infobox` 是**原始 wiki 字符串**（跟在线 API `/v0/subjects` 已解析成 `[{key,value}]` 数组**不同**），
  别名要自己正则抠 wiki（`|别名={ [..] }` / `|别名= 单值` 两种都兼容）。`type`：1漫画 2动画 3音乐 4游戏 6三次元 —— 只留 2。
  压缩包里文件名是 `subject.jsonlines`（脚本用 `unzip -Z1` 自动认，别写死）。实测扫 65.8 万 subject → 3.05 万动画。
  **档里没封面** → 加番时那一次 detail 取 `images.large` 补（也是唯一还碰在线 BGM 的地方，加番才触发、量小）。
- **模糊搜索的坑**：纯 `LIKE '%整串%'` 是「一字不差的子串」，差一字就搜不到（「漫画咖啡厅」找不到「漫画咖啡屋」，而 BGM
  网页能）。改成把查询拆 **CJK 相邻二元组**、命中任一即入选、按命中数排（拉丁词整段不拆）。纯查询层改动、不动索引。
- **部署/运维**：`bgm_index.db` 落 `DATA_DIR`（跟 web.db 一样在部署目录外），是**独立只读**大库、可整体重建。VPS 需装
  `unzip`；cron 每周三跑一次 `scripts/build-bgm-index.ts` 重建（原子替换 → 搜索端按 mtime 变化自动重开、不重启）。本地跑带 `HTTPS_PROXY`。
- **边界**：本周三后刚上架 BGM 的**冷门新番**要等下周档才进索引（当季在播有实时周历兜着）。真需要可加「本地搜不到 →
  退回一次 BGM 在线搜」，冷门才触发、量极小。

### 网页项目放哪（2026-07-13 已定：同仓库 `web/` 子目录，结构隔离）

用户要求**必须在同一个仓库**（不接受另开 git 项目），但**绝不影响 app**。已核对 app
构建配置，确认放一个顶层 `web/` 子目录、且**自带独立 package.json + 独立 node_modules**
就能物理隔离（下面每条都对着实际配置查过）：

- **类型检查扫不到**：`tsconfig.node.json` 的 include 只有 `src/main`+`src/preload`，
  `tsconfig.web.json` 只有 `src/renderer/src`；`web/` 不在任何 include 里。
- **electron-vite 构建碰不到**：main 入口写死 `src/main/*`，renderer 默认根 `src/renderer`，
  不扫 `web/`。
- **打包不会带上**：electron-builder 的 `files` 是白名单（`out/**` + 根 `node_modules/**`
  + `package.json`），`web/` 不在内；web 依赖装进 `web/node_modules`（不是根 node_modules，
  也就不被根 `node_modules/**` 匹配），不会被塞进安装包。
- **根命令零改动**：`web/` 用自己的命令（`cd web && npm run dev`）跑，**根 package.json
  一行都不加** —— app 侧字面上零改动 = 零风险。
- **产物已被忽略**：`.gitignore` 里 `node_modules/` `dist/` `out/` 是全局规则，`web/` 下
  同名产物自动忽略，只有 web 源码进 git（正合「同仓库」要求）。
- **红线：绝不把根改成 npm workspaces** —— 那会动根 package.json、把 web 依赖 hoist 到根，
  反过来影响 app。要的就是「两套 package.json 各装各的、互不相干」。
