## 📖 详细说明

### BGM 限流防御与错误反馈

**目标效果**：

1. 用户在任何 BGM 入口被限流时，前端看到「BGM 触发限流，请等 N 秒再试」
   这种**准确**的提示，能 self-serve 判断"歇会儿再来"，而不是反复点击
   重试加剧限流
2. 周历刷新失败不再装作成功 —— 用户点刷新后如果 BGM 不让拉，要明确
   反馈"刷新失败，仍显示 X 时间的缓存"
3. 一次浏览不会自己把自己限流 —— 详情页连点几个 + 关联追番搜索 +
   周历刷新组合操作时，请求被全局节流到 BGM 能接受的频率

**复现**：

- 关键词搜索：在 SearchDownload 关联追番弹窗里搜一个相对冷门的长关键词
  （比如「好比是最终迷宫前的少年到新手村生活一般的故事」），多页结果触发
  限流后弹「Error: Error invoking remote method 'bgm:search': Error:
  网络请求失败，请检查网络连接后重试」—— 但浏览器能正常打开 bgm.tv 说明
  网络没问题。
- 周历刷新：进 `/calendar` 点右上「刷新」按钮，按钮转一下但缓存时间戳没变
  （比如还是 5/11 20:00），实际是限流被吞了。
- 惩罚窗口 30s ~ 1h 不等，等到自动解除一切正常。

**已经做过的事**（避免重新踩坑）：

- 历史限流防御（commit 54fbe75 / bb39b59 / 2a4d418，已记在 001 的「已经
  做过的事」里）—— 给 bgm.tv 搜索建了三层防御：2.2s 间隔 RateLimiter +
  浏览器指纹 + body "您在 N 秒内只能进行一次搜索" 检测。但这套**只覆盖了
  搜索 HTML 抓取**，`api.bgm.tv` 端点（详情 / 别名 / 周历）完全没接入。
  本阶段的核心就是补 api.bgm.tv 的限速。
- v0.2.0 加的别名回退（commit c7f2cf7）—— 搜索 visible 命中为空时回退
  api.bgm.tv 查别名，**最多 8 条 + 3 连续 miss 早停**。早停虽然在，但
  最坏情况仍可能突发 8 个 API 调用，叠加详情 / 周历就过载。这条 cap 不
  要再放宽。
- 搜索那一层的 `withRateLimitRetry`（限流后等待 + 重试一次）这套机制是
  好的，可以直接复用到 api.bgm.tv。不要重新发明类似的重试逻辑。
- `fetchPage` 当前用 `try/catch` 把所有非 RateLimitError 转成 `null` 返回
  —— 这屏蔽了真实错误。修法是让它把原始 Error 抛出去，由 `searchBgm` 统一
  分类成"limit / network / 5xx / timeout"几种带 cause 的错误类型。**别**
  在 fetchPage 里就开始分类（让它纯透传）。
- BGM 限流惩罚窗口的实际长度 30s 起步，惩罚累积时可到几十分钟。所以
  Layer 1 timing（2.2s 间隔）必须严守，**不要为了"快一点"调低**。
