# 008 - BGM 限流熔断与降级

> 开始：2026-05-31

## 🎯 这个阶段的产物

BGM 被限流后能**自动降温并恢复**，而不是越用越严、长期卡死：触发 429 后停止继续
投喂 `api.bgm.tv` 一段时间（阶梯冷却，最长对齐 BGM 的 24–48h 惩罚窗口），冷却到点
自动半开试探恢复；日常单次搜索的 API 突发更小、连搜也不会一口气吃光分钟级配额。
进一步（第二步）：冷却期改走 `bgm.tv` HTML 降级，让搜索/详情**冷却时仍可用**。

用户视角：正常按中文别名搜资源不再三天两头卡在限流上；真被限了也有明确「冷却中，
约 X 分钟后恢复」提示，而不是一直转圈或反复加重惩罚。

## 📖 详细说明

### 限流根因（已查清，不要再走弯路）

**复现**：按**中文别名 / 简称**（非主标题）连搜几个资源（如搜「魔女的考验」，
其条目主标题是「魔界女王候补生」），很快收到限流；之后连正常搜索也持续被限，
要等很久才恢复。

**已经做过的事**（避免重新踩坑）：
- 已定位主因：**不是请求频繁，是单次操作放大成 api.bgm.tv 突发**。按别名搜时
  HTML 结果无 visibleMatch（`search.ts:507` `matched.length === 0`）→ 掉进别名回退
  → 对前 N 条未命中项逐个 `fetchAliases`（`/v0/subjects/{id}`）= 单次 2–8 个 API 突发。
  用户基本每次搜索都走中文别名 → 几乎每搜必触发。
- 手动浏览网页没事、app 却被限：浏览走 `bgm.tv`（松端点），app 详情/别名走
  `api.bgm.tv`（严端点，另算配额）。两个端点限流桶是分开的。
- 现有 500ms 间隔 + 抖动**只压瞬时速率**，压不住两件事：①时间窗内的**累计配额**
  （连搜几次 = 几十个 API 吃光分钟预算）；②BGM 的**滑动惩罚计数器**（手册 §4，
  24–48h，触发后阈值越来越低）。普通使用本身在持续投喂，计数器一直被顶起来 →
  「一直加剧、一直用不了」。
- 用户已拍板：**候选数降到 4 可以**；**减少分页不可接受**（结果不全没意义），
  所以 L1 只动候选数、不动分页早停。
- 相关位置：`src/main/bgm/api-client.ts`（共享 RateLimiter / RateLimitError / 无自动重试）、
  `search.ts:509` `ALIAS_LOOKUP_LIMIT=8`、`search.ts:519` 连续 3-miss 早停、
  `detail.ts` 每次详情 2 个 API（subject + persons）、`docs/bgm-集成参考手册.md`
  §3 防御层 / §4 惩罚窗口。

### L2 窗口预算

**目标效果**：连续多次搜索时，api.bgm.tv 请求被自动拉开节奏，不会在分钟级窗口内
把配额一口气吃光（从"只压瞬时速率"升级到"也守累计窗口"）。超额请求排队等窗口
滑动，而不是硬发。

**已经做过的事**：
- N 取保守可调常量 —— BGM 未公开确切配额，是启发式，先给稳值再按体感微调。

### L3 熔断 + 软恢复

**目标效果**：被限后不再继续投喂 API、让滑动计数器自然衰减；冷却到点自动恢复；
恢复初期慢速跑一段再回满速，不立刻重蹈覆辙。

**已经做过的事**（动手时不要再讨论的约束）：
- 状态机：`closed`（走 API）/ `open`（限流中）/ `half-open`（冷却到点试探）。
- 触发：API 收到 429（`RateLimitError`）→ open。
- 冷却时长：有 `retry-after` 用它；否则阶梯升级（分钟 → 小时 → 封顶 24–48h，
  对齐 BGM 惩罚窗口）。**必须持久化到磁盘**，重启不能又去捅 API。
- 半开：冷却到点，仅在**下次自然请求**放**一个** API 试探，成功 closed / 又 429 升级
  重 open。**不主动探测**（遵手册「零主动探测」原则）。
- 软恢复：刚 closed 后 min-gap 先放大（约 1.2–1.5s）跑一段再降回 500ms。
- **本阶段分两步**：第一步 open 期间先**优雅降级**（用缓存 / 明确"冷却中"提示），
  把"不再加剧、自动恢复"跑通；第二步再补 HTML 降级让冷却期也能用。
- HTML 降级解析较脆 → **HTML 优先 + 字段缺回退 API**，放第二步，别在第一步纠缠。
