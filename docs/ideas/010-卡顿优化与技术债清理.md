# 010 - 卡顿优化与技术债清理

> 开始：2026-06-07

## 🎯 这个阶段的产物
把多轮性能优化 / 功能重构攒下的真实技术债清掉，应用像"一次写成、对症下药"：

1. **不卡**：进设置返回 / 切 tab / 从 homework 进 MyAnime 不再有几百 ms ~ 1s 的
   冻结，切换后界面立刻可见、交互跟手。渲染侧（写盘下沉 idle、memo +
   content-visibility）+ 主进程侧（同步盘 I/O 异步化）双管齐下。
2. **报错可查**：主进程所有报错落盘到 `main.log`，不再被 catch 静默吞掉；数据
   加载 / 抓流 / 更新 / 配置保存失败留痕，"线上出问题无从查起"消除。
3. **健壮**：若干边界 / 资源泄漏修掉。

同步 JSON I/O 收敛进一个 `JsonStore`、报错收敛进一个 `logger`，不为去重而去重。
Windows 性能弱、同样的卡顿放大 2-3 倍，是本阶段的验收基准机。

## 📖 详细说明

### localStorage 同步写盘下沉到 idle（渲染侧）
**目标效果**：任何追番 / 推荐的增删改，UI 立刻响应，序列化 + 写盘不再阻塞
渲染线程；连续多次改动合并成一次写。

**复现**：追番条目较多（上百）时，编辑任意一条的状态 / 集数，能感到一下顿挫；
切 tab / 进设置返回同理。Windows 放大 2-3x。

**已经做过的事**（避免重新踩坑）：
- 根因：`animeTrackStore` / `recommendationStore` 的 `persist()` 每次 mutate
  都同步 `JSON.stringify(整张表)` + `localStorage.setItem`，再通知订阅者 ——
  追番上百条时单次 stringify 几百 KB，全跑在渲染线程上。
- 落地：新增 `src/renderer/src/utils/deferredStorage.ts`，`requestIdleCallback`
  合并写盘（同 key 多次 schedule 只留最后一个闭包，闭包在 flush 时才读 store
  抓最新值），`pagehide` / `beforeunload` / `visibilitychange=hidden` 兜底 flush
  防丢数据。两个 store 的 `persist()` 改成**先同步 notify 订阅者（UI 立即更新）
  再 schedule 写盘**。
- **不要回退**成"每次 mutate 同步写 localStorage" —— 那就是这次要消除的卡顿源。
- 延后写盘的前提：没有任何代码在 mutate 后**立即同步回读** localStorage 那个 key。
  现状满足：store 只在首次 `ensure()` 读一次，之后走内存缓存；WebDAV push 用的
  是内存里的 `tracks`，不读 localStorage。以后若新增"写完马上读盘"的逻辑要重新评估。

### AnimeSyncBar dirty 检测防抖（渲染侧）
**目标效果**：MyAnime 顶部「未同步」chip 的状态计算不再每次 store 通知都全量
stringify 整张表，列表交互不被它拖卡。

**已经做过的事**（避免重新踩坑）：
- 原本 `const currentSnapshot = useMemo(() => snapshotOf(tracks, recommendations), [...])`
  —— 但 store hook 每次通知都返回**新数组引用**，useMemo 依赖每次都变、memo 根本
  不命中，于是每次 mutate / 重渲染都同步 stringify 整张表（几百 KB）。
- 改成 200ms 防抖的 `useEffect`，结果写进 `localDirty` state。`localDirty` 只驱动
  chip 的启用态，延迟 200ms 出结果对用户无感。
- 真正 push / pull 时仍直接 `snapshotOf(...)` 算（用户主动触发，不是热路径，不用动）。

### MyAnime TrackRow memo + content-visibility（渲染侧）
**目标效果**：单条追番编辑 / 搜索框输入时不再整列重渲染；初次进入 MyAnime 的
首屏排版只算可见的几行，上百条追番也能秒开。

**已经做过的事**（避免重新踩坑）：
- `TrackRow` 用 `memo()` 包裹：它只依赖 `track` 这一个 prop，且 `upsert` 只替换
  被改那一条 track 的对象引用（其他 key 对象身份不变），`filtered` 又是纯
  `.filter/.reverse/.sort` 不克隆元素 —— 所以单条编辑 / 搜索触发 MyAnime 重渲染时，
  未变行的 prop 引用不变，memo 跳过，重渲染从 O(N) 降到 O(1)。
- 行根 div 加 `content-visibility: auto` + `contain-intrinsic-size: auto 140px`：
  浏览器跳过视口外行的 layout/paint，首屏排版开销大降。
- **注意**：content-visibility 只省浏览器侧 layout/paint，React 仍会为全部行建 DOM
  节点。它和 memo 互补：memo 省 JS 重渲染，content-visibility 省浏览器排版。
- **没有**上虚拟列表 / 窗口化 —— 用户明确要求别过度优化堆复杂度。这些行是变高的
  复杂卡片，虚拟化要处理动态高度 + 滚动锚定，成本高、易出 bug。先用上面两条轻量
  手段，Windows 实测仍明显卡再考虑（见 TODO 末条），且要先测后做、别预先动手。

### JsonStore（主进程 JSON 持久化助手）
**目标效果**：主进程任何 IPC 不再因同步 `readFileSync/writeFileSync` 阻塞整个
事件循环 —— 同步 I/O 期间所有 IPC 都卡着，renderer 切 tab 正等这些返回。这是
切页卡顿的**另一半根因**（渲染侧之外）。

**已经做过的事**（避免重新踩坑）：
- 新增 `src/main/shared/json-store.ts`。唯一设计：`normalize` 必填（解析+容错+
  默认值一肩挑）；**内存为唯一权威值**（读瞬时不碰盘）；写同一 tick 内
  `setImmediate` 合并成一次原子落盘（`.tmp`+`rename`）；`before-quit` 同步兜底
  防丢。语义对标 renderer 的 `utils/deferredStorage.ts`。
- 迁移：`ipc/system.ts`（cache:get / download:load+save-state / history /
  set-setting）、`library/api.ts`（getPaths/setPaths/getEntries/setEntries —
  **签名不变**，内部 `current()` 同步读内存，调用方零改动）、`bgm/api-circuit.ts`
  （熔断器热路径 `current()` 读内存、异步落盘）。
- **刻意没动**已是异步的 `mail/config.ts`（含加密）、`ipc/webdav.ts`（含一次性
  路径迁移）、`aowu/url-resolver.ts`（含 TTL+防抖）、`bgm/calendar.ts`（含 TTL
  信封）—— 它们没有同步 I/O 性能债，强行套 JsonStore 是为去重而去重、还会动到
  加密 / 迁移 / TTL 逻辑徒增风险。`calendar.ts` 只去掉冗余 existsSync。
- 红线：不要把这些再改回同步 `*FileSync` —— 那就是这次消除的卡顿源。

### logger（持久化日志）
**目标效果**：打包后的 app 里报错有处可查 —— 这是"报错被吞"的根因（之前 36 处
`console.error` 在 packaged 下无任何落盘出口）。

**已经做过的事**（避免重新踩坑）：
- 新增 `src/main/shared/logger.ts`：写 `app.getPath('logs')/main.log`（macOS
  `~/Library/Logs/MapleTools/`、Windows `%APPDATA%/MapleTools/logs/`），按大小
  轮转（main.log + main.1.log 两代）。
- `initConsoleCapture()`（main 启动调一次）接管 `console.error/warn` →
  **保留原控制台输出的同时自动落盘**，不必逐处改 36 个站点、未来的也自动覆盖。
  关键：在 patch 前抓住原生 console 引用，logger 自身回显用原生引用，**否则递归 /
  重复落盘**。
- renderer 未捕获错误 / unhandledrejection 经 `log:error` IPC 转发汇入同一文件
  （`main.tsx` 挂全局监听，preload 暴露 `window.systemApi.logError`）。
- 设置→关于「打开日志目录」按钮（`shell.openPath(app.getPath('logs'))`）。

### 停止静默吞错误
**目标效果**：真实失败留痕可查、数据路径"坏一条不丢全部"，但**不改控制流、
不加自动重试**（违反 BGM 集成红线）。

**已经做过的事**（避免重新踩坑）：
- 新增 renderer `utils/reportError.ts`：控制台 + 转发主进程落盘；`backupCorrupt`
  在 localStorage 解析失败时把坏数据备份到 `${key}.corrupt` 再返回（不静默丢、
  也不让坏 JSON 永久挡住后续写入）。
- `animeTrackStore` / `recommendationStore` 读取：整份解析失败不再静默清空，改
  备份 + reportError。`downloadStore.init`：统计被丢弃任务数并 reportError。
- updater 后台静默检查失败 / autoUpdater error：空 catch → logError（不弹错打扰，
  只留痕）。girigiri 抓流 30s 超时 / 页面加载失败：静默 done(null) → logError。
  Settings webdav/mail 保存失败：空 catch → reportError。
- **坑**：`reportError` 与 DOM 全局 `reportError(error)`（只 1 参）同名，不 import
  会静默解析到全局版、TS 报"Expected 1 arguments"。各处必须显式 import 覆盖。

### 边界与资源

- girigiri `parseM3u8`：20MB 体积上限 + master playlist 嵌套深度封顶（防损坏/恶意

- aowu url-cache：落盘时剔除过期项（常驻不重启会话里原本只增不减）。
- `ipc/system.ts` connectivity 计时器改在全失败路径也清除。
- **复核后跳过的 agent 误报**（别再去"修"）：calendar-mailer 的
  `Number(x)||DEFAULT` 本就挡住 null/NaN/0、`handleOnce` 回调不会抛；
  speed-tracker 由 `forgetTask` 在 cancel/done 清理、按任务数有界；SearchDownload
  搜索是显式提交不是 per-keystroke、不用防抖；`useAnimeTrack(ByBinding)` 靠稳定
  对象身份让 React bail-out、单条 mutate 不会让所有卡片重渲染；下载队列被误报为

### 冷启动卡顿：媒体库扫描（最硬的一块，踩坑全记）
**现象**：每次启动后约十几秒，做什么都卡 ~2s（切 tab、甚至**拖不动窗口**），等
`startup:silent-scan` 打印（扫描结束）之后立刻顺。`probeToPaint` 量到的 `nav:` 才
124ms，跟体感的 2-3s 对不上。

**根因（一层层挖出来的，别再走回头路）**：
1. `probeToPaint` 只量「路由变更 → 下一帧绘制」，抓得到同步渲染、**抓不到首帧之后
   异步填充的封面**。封面慢才是体感卡，不是渲染慢。
2. 启动时 `app.whenReady()` 立刻 `runSilentScan()`（全量遍历媒体库）。MyAnime 首次
   挂载又给每个可见行发 `cacheCover` IPC。两者抢同一批资源 → 封面命中(本应毫秒)被
   拖到 1.2s，一批一起返回（事件循环被占死的特征）。
3. **并发救不了**：把遍历并发从 1 开到 8 几乎不提速（受单一资源吞吐限，不是 IO 延迟限）。
4. **`setImmediate` 让一拍没用**：它只让出 JS 主线程，**不让 libuv fs 线程池**（默认 4 条）
   喘息，扫描排队任务照样霸着线程池 → 封面读盘还是排在后面。
5. **最终根因 = 物理磁盘被打满**：「拖不动窗口」是 OS 级卡顿，只可能是物理盘 100%。
   慢盘 / 机械盘上全量扫描十几秒的 I/O 把整机拖卡。**线程 / 线程池都治不了物理盘饱和。**

**已经做过的事（按生效顺序，避免重新踩坑）**：
- 探子：`src/renderer/src/utils/probe.ts` 已有；本阶段加 `startup:silent-scan`
  （`index.ts`）、`cacheCover` 慢调用（`bgm/cover-cache.ts`，gated >100ms，hit/dl 区分）。
  **是这两个探子的数据**（封面全 `hit` 却 800-1200ms、且齐刷刷返回）锁定了"饿死"而非"慢"。
- 启动重活延后：`runSilentScan` + `startLibraryWatch` 推迟到 `app:renderer-ready` 后
  ~1.2s（5s 兜底），让首次交互 / 封面先抢到资源（`index.ts`）。
- 封面目录只建一次：`cover-cache.ts` 用共享 promise，去掉每张封面冗余 `mkdir`。
- `UV_THREADPOOL_SIZE=16`：`shared/uv-bootstrap.ts`，**必须是 `index.ts` 第一个 import**
  （线程池首次用到时按当时的值创建，之后改无效）。
- **扫描搬进 worker_threads**：`library/scan-core.ts`（纯 Node 遍历逻辑）+
  `library/scan-worker.ts`（worker 入口）。worker 有独立事件循环 + 独立 fs 线程池，
  扫描的 JS 与 fs 全在那条线程 → **主线程 + 主线程池全程空闲**，封面 starvation 消除
  （探子实测 `cover:* hit` 不再刷屏）。`electron.vite.config.ts` 把 worker 作为 main 的
  第二个入口打包到 `out/main/scan-worker.js`；`package.json` `asarUnpack` 它 + chunks。
  worker 起不来时 `scanLibrary` 退回主线程 `syncLibrary` 兜底，功能不丢。
- 扫描「低 I/O 优先级」温柔限速：`scan-core.ts` 的 gate 并发压低 + **真 sleep**（不是
  setImmediate）。worker 隔离了线程，sleep 再让物理盘留空窗给前台 —— 双管才不卡整机。
- **目录 mtime 增量同步（终极方案，对标网盘）**：NTFS 在目录直接子项增删改名时更新
  该目录 mtime。持久化每目录 mtime（`library_dir_index.json`，不动 `LibraryEntry` 结构），
  启动只 stat 比对：未变目录 → 复用缓存条目、连文件都不读、用已知子目录递归连 readdir
  都省；变了 / 新目录才深扫。整库没动时启动只剩"每目录 1 次 stat" → **亚秒级、全自动**
  （离线改动也被父目录 mtime 变化带出来，无需手动刷新）。`syncLibrary` 在 `scan-core.ts`，
  启动 / 手动刷新 / 加路径都走它；删路径只是把那段索引剔掉（`removePath`），不扫描。
  **首次 / 索引为空** → 每目录都算"变了" → 自然退化成一次全量扫描并把索引建起来（一次性，
  之后都增量）。

**关键认知（写给未来的自己）**：
- "全异步就不卡"不成立：再多异步也要排队过**单一稀缺资源**（主线程 / 4 条 fs 线程 /
  一块物理盘）。卡的根因是**争用**，解法是**减少 / 错峰 / 隔离争用**，不是"再异步一点"。
- 慢盘上"扫得更聪明"治标，"**根本别每次全量扫**"才治本（网盘思路：持久索引 + 只处理 delta）。
