## 📖 详细说明

### 封面本地化（方案 C）

**目标效果**：列表封面走本地文件，离线可看，不每次重新拉 lain.bgm.tv。

**已经做过的事**（避免重新踩坑）：

- **封面 ≠ 限流原因**：封面走 `lain.bgm.tv` 图片 CDN，跟 `api.bgm.tv`（搜索/
  详情/周历，限流就在这）是**两套独立限流账本**。本地化的真正价值是
  性能 + 离线，**不是**为了缓解限流。所以 `cover-cache.ts` 不接 RateLimiter
- **方案 C = 所有封面都本地化**（不只手动加的）：BGM 加追番的两个入口
  （AnimeInfo「Track this anime」/ AnimeCalendar「+追番」）upsert 后都
  `void cacheCoverFor(bgmId)`。用户决策：手动 / 非手动行为统一
- **时序**：先 upsert（cover=原 URL，UI 立即有图）→ 异步下载 → 成功后
  再 upsert（cover=archivist:// 本地路径）。失败保留 URL，**不重试**
  （符合 BGM 集成参考手册 §3「失败不试探不重试」）
- **archivist:// 已有协议**（main/index.ts:96）能服务任意本地文件，封面
  存 `userData/covers/{key}.{ext}` 直接复用，不用新协议
- 幂等：cover 已是 archivist:// 或本地已有同 key 文件 → 不重复下载

### 手动添加

**目标效果**：限流时不靠 BGM 搜索也能加番，且填了真 bgmId 后限流恢复
无缝衔接（详情页认出已追番 + 自动补全）。

**已经做过的事**（避免重新踩坑）：

- **bgmId 设计**：用户优先填真的（从 bgm.tv 条目 URL 拿，限流不影响浏览器
  访问 bgm.tv 主站）→ 用真 id。填不出来 → 留空 → 系统给 `-Date.now()`
  负数兜底（真 BGM id 都是正整数，亲不冲突）。用户决策见 006 起始
  AskUserQuestion
- **为什么填真 id 就无缝**：track store（maple-anime-tracks-v1）和 detail
  cache（search_cache.json 的 bgm_detail_cache）**物理隔离**。手动加只写
  track store 的最少字段，detail cache 那边是空的。详情页 `useAnimeTrack(bgmId)`
  按 bgmId 查 store → 命中手动 track → 显示已追番；`getCachedBgmDetail(bgmId)`
  miss → fresh fetch → 拿全量。手动加的最少数据**不会污染** detail cache
- **入口位置**：MyAnime 各类目 tab 顶部（动画/漫画/小说），不是 AnimeInfo
  限流出错屏。限流不限流都能用，不是限流专用。用户决策见 AskUserQuestion
- **最少必填**：标题 + 类目（默认 = 当前 tab）。bgmId / 中文标题 / 封面 /
  总集数都可选。状态默认 plan，集数 0
- **重复 bgmId 拦截**：填的真 bgmId 已在追番列表里 → 报错不让加（避免覆盖）
