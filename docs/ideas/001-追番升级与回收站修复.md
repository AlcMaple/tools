# 001 - 追番升级与回收站修复

> 开始：2026-05-12

## 🎯 这个阶段的产物

围绕"番剧追踪 / 周历 / 跳转"的三个动漫功能全套落地，并且文件资源管理器
的删除（移到回收站 / 永久删除）在 Windows 上面对被占用的文件夹也能干净
完成，不再抛出「PowerShell 结果解析失败」白板错误。

### 1. 追番记录（MyAnime 汇总页 + AnimeInfo + WebDAV）

- **入口收敛在两处**：① AnimeInfo 详情页左栏「Track this anime」toggle；
  ② SearchDownload 卡片角的「关联追番」按钮（点开 mini BGM 搜索弹窗，
  选条目即绑定）。关联成功后，SearchDownload 卡片左上角出现 `EP n` 角
  徽章，代表"这部已经在追了，看到第 n 集"。
- **MyAnime 汇总页**展示所有追番条目，每行：海报 · 标题 · 状态 segment
  · 集数控件 · 已绑定的源 chip · 编辑按钮 · 解除追番按钮。备注 UI 没有
  做（用户明确不要），store 里 notes 字段只为兼容老数据保留。
- **状态 4 档**：`想看 / 在看 / 看完 / 搁置`，segment 风格切换，变更
  立即落 store。
- **集数控件三种交互**：① `−` 按钮回退一集；② 中间数字框可直接键入
  目标集数（onBlur 提交）；③ `+1` 按钮前进一集。`episode > totalEpisodes`
  时自动 clamp 到上限，不会出现"看完总集数还能再 +1"。
- **集数显示约定**：`current / total`。`episode = 0` 时 chip 上的集数段
  完全不渲染，避免"从头开始"之类的冗余字样。
- **总集数 = 双态字段**（最新落地）：BGM 对长寿番 / 特别篇 / 季番初期
  会返回 `eps: 0`，这种番以前会一直卡在 `0 / ?`——`?` 含义模糊，用户
  不知道是"BGM 没数据"还是"番剧没完结"。现在彻底重设：
  - **总集数未填时显示「连载中」徽章**（primary tint + 同步图标），
    一眼就能看出这部番"还在更新中"，不再用 `?` 这种含混字符。
  - 点击「连载中」徽章 → 切换到可编辑数字框，用户填完整集数（番完结
    了 / 知道最终集数了就填进去，比如 12、24、1000）。
  - 数字框留空提交 → 退回「连载中」徽章状态（仍然意味着"还在连载"）。
  - 改小了总集数会自动 clamp 当前集数（store 内部 normalize）。
- **设计取舍**：徽章/数字"两态"对应的语义：
  - **「连载中」**：番剧还在更新、或用户暂时不想填具体集数。这是新番
    和长寿番的默认态。用户可以正常用左边的数字框记录"我看到第 N 集"。
  - **具体数字**：番剧已完结 / 用户已知最终集数。比如完结的 12 话季番、
    剧场版 1 话、柯南这种填用户预期值（比如 1500）也行。
    填的值是"最终总集数"语义，不是"当前连载到第几集"——后者每周都要
    改一次 UX 不划算；前者只在作品基本完结或预期明确时填一次就好。
- **MyAnime 行排序按插入顺序**（store Map 迭代顺序），不按 updatedAt。
  原因：改任意 chip / 集数都会刷新 updatedAt，按它排会让正在编辑的行
  突然跳到顶部，体验糟。
- **WebDAV 同步独立**（最新落地）：追番数据从原来塞在 homework.json 里
  的 `tracks` 字段彻底拆出来，单独存到 `anime.json`。两边各自维护 rev /
  冲突检测 / 确认弹窗，互不影响。具体：
  - 用户在 Settings 里只配一个 base 文件夹（`MapleTools`），主进程拼出
    `MapleTools/homework.json` 和 `MapleTools/anime.json` 两个文件路径。
  - 老配置（`MapleTools/homework.json` 这种完整文件路径）首次加载时自动
    剥成 base folder 写回磁盘，用户无感。
  - HomeworkLookup 顶部的同步 chip 只管 homework.json；MyAnime 顶部
    `AnimeSyncBar` 自己一份 chip + 上传/下载/确认弹窗，只管 anime.json。
  - 改一个 chip 进度只 push anime.json（几 KB），不会把整个阵容数据库
    带上一次。
  - 跨设备并发安全：A 设备改追番、B 设备同时改阵容，互相不覆盖（旧
    方案下 push 必须先 pull 整个 blob、覆盖式写回，并发会丢数据）。
  - 升级迁移兜底：anime.json 在远端不存在时（首次升级），AnimeSyncBar
    会自动回退去读老 homework.json 的 `tracks` 字段，拉取时弹窗顶部
    会有「从老数据迁移」的提示带 ✓ tertiary 色块，用户确认后老数据
    迁过来。push 后 anime.json 就建立起自己的 rev 链，老 homework.json
    里残留的 tracks 字段被新版本无视。

### 2. 多源绑定（一番多源）

- **一部番可同时绑定多个观看源**。源分两类：
  - **内置三源** `Aowu / Xifan / Girigiri`：必须走搜索流程绑定（点
    MyAnime 行尾 / SearchSourceModal 的「+ 搜 Aowu / Xifan / Girigiri」
    入口），结构性数据（id / 路径），没有用户能改的字段，
    EditBindingsModal 不显示。
  - **用户自定义源** `Custom`（旧数据可能是 `Bilibili`）：走
    AddBindingModal，标题 + URL 两个字段，自由格式。
- **AddBindingModal 极简**：只问"显示在 chip 上的标题"和"URL"，移除了
  早期"B 站 preset / 自定义"两段切换 —— 用户反馈"没必要分类型",
  统一为 Custom，sourceTitle 兼任 chip 标签。
- **EditBindingsModal 统一管理用户加的链接**：点 MyAnime 行尾「编辑」
  按钮打开，列出所有 user-added bindings，每行可改标题 / 改 URL /
  点 ✕ 标记删除，最后「保存」一次性 commit。校验：URL 必须 http(s)
  前缀、同 URL 不能重复。取消 / 关弹窗丢弃所有改动。
- **删除入口物理隔离**：WatchHere chip 本身永远是纯跳转 `<a>`，没有
  任何删除控件。删除集中在 EditBindingsModal 里。早期尝试过"chip
  内联红色 X"，用户明确反对（"用户很容易点错"）。
- **关联后可补绑其他源**：MyAnime 行尾「+ 搜 X」打开 SearchSourceModal
  （X = 当前未绑的内置源），用户自己选用哪个搜索结果补上去——一番一源
  不再是硬约束。搜索流程走 useSourceSearch hook 复用 SearchDownload
  的缓存层 + Aowu 流式 + 验证码弹窗 + reqId 串行化（React 18 strict
  mode 双 fire 会出现"结果先闪现再消失"的 bug，靠 startedRef +
  reqIdRef 双 guard 防住）。

### 3. 番剧周历（/calendar）

- **默认显示本季更新**，周一到周日 7 列网格，纵向滚卡片，无水平滚动
  （默认 1280px 窗口刚好放得下，靠 grid-cols-7 + px-6 自适应，无 min-w）。
- **缓存 14 天**（早期是 24h，用户要求拉长），右上「刷新」按钮强制
  update=true 绕过缓存重新拉一次。
- **当日列高亮**（颜色 / 边框），不再有"TODAY"角标字样——用户审美偏好,
  反正高亮一眼就知道是今天。
- **滚动时置顶的是"周一-周日 chip + 缓存/刷新按钮"两行 sticky bar**,
  不是页面标题。早期单行布局（chip 和刷新挤一行）会让 chip flex-1
  比 cards 行的 grid 列窄，上下对不齐；两行设计是为了像素级对齐。
- **卡片标题强制 2 行高**（min-h），保证每列每张卡片高度一致，不会
  出现长标题撑高这列、短标题留白那列的参差。
- **卡片 hover 出 WatchHere chip**，可以直接从周历跳进观看页。

### 4. 跳转观看（WatchHere chip）

- **三处统一部署**：① AnimeInfo 左栏（variant=row）；② MyAnime 行尾
  （variant=inline 紧凑横排）；③ Calendar 卡片 hover（variant=row）。
- **chip 显示**：图标 + 源名（Custom 显示 sourceTitle，Bilibili 显示
  sourceTitle 或 'B 站' fallback，内置源显示 source enum 名）+ `ep N`
  或 `ep N/Total`。`episode = 0` 时省略 ep 段。
- **点击在外部浏览器打开源详情页**。不为每个源调 watch_info 算"ep N+1
  的具体播放页 URL"——那要全量抓一遍，太重；直接开源详情页，让用户
  自己选下一集。chip 上的 `ep N` 起到进度提示作用。
- **Aowu URL 修复**（这阶段一个硬骨头）：早期合成的 `aowu.tv/v/14`
  是后台 id 路由，不是用户可访问页，点开是报错页。现在 chip 跳的是
  `aowu.tv/w/{token}#s={source_id}&ep=1` 用户播放页，通过 Aowu 内部
  的 `route-tokens` action 反向映射拿 token + bundle('video') 拿
  source_id，并行调，组装出真正能打开的播放页。
- **历史 binding 静默 backfill**：早期绑的 Aowu binding 只有 `/v/{id}`
  形式的 sourceKey、没 sourceUrl。MyAnime 渲染时
  `useAowuShareUrlBackfill` 每个 session 试一次 resolveShareUrl，
  成功后通过 store.subscribe 原地刷新 chip 的 href——用户无感知，下次
  点击就是对的 URL；失败留到下次启动再试，不弹错误吐司。

### 5. File Explorer 删除（Windows 占用文件夹）

- **「移到回收站」走两阶段 + 用户中段确认**：
  - **Stage 1（整体送回收站，5s 窗口）**：IFileOperation 直送。绝大多数
    普通文件这步就成，toast 显示「已移到回收站」（可附带"自动关闭 N 个
    占用程序"，如果清障碍那步杀了进程）。
  - **Stage 1 失败 → 弹窗让用户决定**，**不自动**进 Stage 2。弹窗内容：
    解释接下来的操作是"逐个把文件 / 子目录单独送回收站、最后送空根目录"
    （Stage 2 分片回收）；说明回收站里会看到散件而不是一个完整文件夹,
    但内容仍完整可还原（回收站全选 → 右键还原能重建目录结构）。两个按钮:
    「继续走分片回收」/「取消」。
  - **Stage 2（分片回收，用户同意后）**：递归枚举整棵树，文件 → 空目录
    （深度倒序）→ 根目录，逐个 IFileOperation 送回收站。成功后 toast
    强提示"已分片送入回收站，回收站里是散件，全选→还原可重建结构"。
    Stage 2 是 AV/Defender 拦截整目录移动的根因解法（杀软盯整目录移动,
    不盯单点操作）。
  - **用户取消 = 终止流程**，**不**自动 fallback 到永久删除。用户要永久
    删除得自己回去点「永久删除」按钮。这是用户明确决定，跟旧版"失败弹窗
    给「改为永久删除」按钮"的反模式划清界线。
- **「永久删除」UX 不变**：底层走 recycle 工具的 `--purge` 模式
  （Remove-Item → `rd /s /q` → robocopy /MIR 三级 fallback，几乎一次必成）,
  二次确认弹窗保留，toast 仍然显示「已永久删除」+ 可选的"自动关闭 N 个
  占用程序"。
- **UI 仍然只有「移到回收站」+「永久删除」两个顶层按钮**，不开第三个
  「强制删除 / 分片回收 / 跳过 AV」入口（用户明确否过自动 fallback,
  这条约束跨多个迭代都成立）。
- **底层依赖** `docs/Windows删除回收站/recycle.js` + `recycle-helper.ps1`,
  作为一份验证过的两阶段方案（README 是集成参考文档，记录了所有踩过
  的坑和最终策略）。退出码契约：`0` = 整体成功；`4` = 分片成功（必须
  提示用户）；`1` = 失败（弹错误，由用户决定下一步，不自动 fallback）。

### File Explorer 删除：PowerShell 结果解析失败

**目标效果**：Windows 下删除被占用 / 系统保护的文件夹时：
- 能删的——杀进程 + takeown + icacls 自动尝试，最后 SendToRecycleBin
  或 Remove-Item 成功，toast 显示「自动关闭 N 个占用程序」
- 删不掉的——弹窗「原始错误」段是真实可读的失败原因（"Access is
  denied" / "Cannot find path" 之类 Windows 原文），而不是
  `SendToRecycleBin: PowerShell 结果解析失败` 白板字串

**复现**：Windows 下选 `C:\Users\<user>\Downloads\maple-tools-20260511-1748`
（应用自身的安装/解压目录，正在被 Maple Tools 进程占用），点删除：
- 「移到回收站」失败弹窗 原始错误 = `SendToRecycleBin: PowerShell 结果解析失败`
- 「永久删除」失败弹窗 原始错误 = `Remove-Item: PowerShell 结果解析失败`
- 但用户在资源管理器里能看到文件**真的被移进了回收站**——也就是说操作
  其实成功了，是 parser 误判成失败 → 白板错误

**根因**：Chinese Windows 上 PowerShell 默认输出编码是 OEM（GBK / CP936），
Node 这头按 UTF-8 解码 stdout。两边对不上 → 中文（takeown / icacls 的
"成功"/路径里的中文等）变成无效 UTF-8 字节序列 → JSON 字符串结构破损 →
JSON.parse 抛错 → `parsePsDeleteResult` 走 catch 分支返回"结果解析失败"。

**已经做过的事**（避免重新踩坑 / 当前生效的实现）：
- 3ad3e9e 已经把删除统一成「单一作用」模型：trash() / permanentDelete()
  内部自动 kill 进程 + takeown + icacls，UI 上只有这两个顶层按钮，没
  「强制删除」/「升级永久删除」之类的中间态。修 bug 不要开第三个按钮
- 用户明确否过自动 fallback（弹窗里给「改为永久删除」按钮）：失败弹窗
  只展示信息，不提供"试试更激进的方式"入口
- **PS-stdout 解耦方案**（当前实现，从"强制编码 + 兜底"升级到"砍掉这条
  通路"，因为编码兜底永远不够彻底）：
  1. **职责拆分**：PS 只做"清障碍"（Restart Manager 杀进程 + takeown +
     icacls 改权限）。实际删除由 Node 干（`fs.rm` 或 `shell.trashItem`）。
     PS 不再做删除，也不再"包打整个流程然后回报结果"。
  2. **数据走临时文件**：杀进程列表通过
     `[System.IO.File]::WriteAllText($tempPath, $json, UTF8Encoding($false))`
     直接写 UTF-8 字节到 `os.tmpdir()` 下唯一临时文件，Node 用
     `readFile(path, 'utf8')` 读。整个 string ↔ UTF-8 bytes 转换在 PS 和
     Node 进程内部完成，**完全绕过 console / OEM codepage**——这才是
     根因消除而非缓解。临时文件 `try/finally` 清理。
  3. **PS stdout 一字不读**：`runWindowsPowerShellSilent` 不读 stdout 不
     检查 exit code，takeown / icacls 都 `*> $null` 丢输出。PS 进程是
     "干完活就算"，怎么算成败由 Node 看文件系统。
  4. **`existsSync` 是唯一真理**：删完看 target 还在不在。在 = 抛错（带
     `Remove-Item:` 或 `SendToRecycleBin:` 前缀，让 renderer 的 friendlyError
     能按 "access is denied" / "拒绝访问" / "正在使用" 关键词归类）；
     没了 = 成功，返回 killed 列表。
- 失败时 message 形如 `Remove-Item: 操作未生效，文件夹仍然存在（可能被
  系统驱动 / 内核 / 杀软占用）` —— 真到这步基本就是 Restart Manager 也
  找不到的奇葩占用（杀软驱动 / 文件系统过滤驱动等），用户重启 Windows
  就能搞定，不需要我们再加复杂逻辑去硬抗
- **方案局限**（后来发现的根因，触发下一条 todo 的原因）：PS 解耦解决了
  "stdout 编码炸 → 白板错误"这类**报错** bug，但没解"AV/Defender 拦截
  整目录的 IFileOperation 移动到 `$Recycle.Bin\`"这一类**功能性**失败。
  Windows 上 IFileOperation 本质是 MoveFileW，杀软盯的就是"整目录树移动"
  防勒索软件批量加密后转移。重启 Windows 释放 watcher 句柄能临时绕开,
  但用户体验是"刚跑过应用的项目目录 / node_modules 老删不掉"。要根除得
  上"分片回收"的方案——逐个文件 / 空目录单独送回收站，AV 不盯这类单点
  操作。这条路由下一个 todo 接管

### File Explorer 删除回收站重构（recycle.js 两阶段方案）

**目标效果**：

1. **「移到回收站」走两阶段 + 用户中段确认**
   - Stage 1（5s 窗口）整体送回收站。绝大多数情况一次成，toast「已移到
     回收站」，结束。
   - Stage 1 失败 → 弹窗解释 Stage 2 会做什么，让用户决定是否继续。
   - 用户选「继续」→ Stage 2 分片回收。成功后 toast 强提示"回收站里是
     散件，全选→还原可重建结构"。
   - 用户选「取消」→ 流程终止，**不**自动调永久删除。用户要永久删要
     自己点回那个按钮。
   - Stage 2 也失败 → 弹错误，由用户决定下一步（重启 Windows / 永久
     删除 / 取消），**不**自动 fallback。

2. **「永久删除」UX 不变**，底层换成 recycle 工具的 `--purge` 模式
   （Remove-Item → `rd /s /q` → robocopy /MIR 三级 fallback）。原有的
   二次确认弹窗、toast、错误处理都不动。

3. **UI 顶层按钮只有「移到回收站」+「永久删除」两个**，跟旧版一致。
   弹窗是 in-flow 决策点，不是新增的常驻入口。

**已经做过的事**（避免重新踩坑）：

- 底层依赖 `docs/Windows删除回收站/recycle.js` + `recycle-helper.ps1`
  这套用户已经验证过的两阶段方案，README 是集成参考。Stage 2 分片回收
  是绕过 AV/Defender 拦截整目录移动的根因解法，杀软盯的是"整目录树
  重命名 / 移动"，不盯"单文件送回收站"。
- 退出码契约（README 强调"集成必读"）：
  - `0` 整体成功 → toast「已移到回收站」
  - `4` 分片成功 → **必须**给用户强提示"回收站里是散件，可全选→还原"
    （README 单独一节强调这点）
  - `1` 失败 → 抛 Error，由调用方决定 UI
  - `already-absent` → 静默成功
- 旧版「PS 解耦 + existsSync 验」方案（上一条 todo 落地的）是中间方案,
  根除了 stdout 编码炸点但没解 AV 拦截。新方案上线后旧的 `prepareForDelete`
  / `trashItem` / `permanentDelete` 函数全部废弃 —— 不要留"双轨"。
- 用户明确否过的反模式（跨多个迭代都成立）：
  - 失败弹窗里给「改为永久删除」按钮（自动 fallback）
  - UI 加第三个顶层按钮（「强制删除」/「升级永久删除」/「分片回收」之类）
  - chip 内联红色 X / 一键删除（容易点错）
- 用户原话约束：「就算用户点击取消也不要自动调用永久删除，最终还是由
  用户来手动点击永久删除来做处理」—— 这条决定了 Stage 1 失败弹窗的
  「取消」按钮的语义。
- 集成形式选择是动手时决策（把 recycle.js 改写成 TS 进 main/、作为 .cjs
  require、或者保留为 spawn 子进程），三种都可行，README "模式 A" 的
  示例代码是 require 形式。
