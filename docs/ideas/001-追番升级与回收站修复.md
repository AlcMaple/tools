## 📖 详细说明

### File Explorer 删除：PowerShell 结果解析失败

**目标效果**：在 Windows 下删除被占用 / 系统保护的文件夹时：
- 能删的——杀进程、takeown、icacls 等内部自动尝试，最后 SendToRecycleBin
  或 Remove-Item 成功，toast 显示「自动关闭 N 个占用程序」
- 删不掉的——弹窗里的「原始错误」段是真实可读的失败原因（比如
  "Access is denied" / "The system cannot find the file specified"），
  而不是当前看到的 `SendToRecycleBin: PowerShell 结果解析失败` 或
  `Remove-Item: PowerShell 结果解析失败` 白板字串

**复现**：Windows 下选 `C:\Users\<user>\Downloads\maple-tools-20260511-1748`
（应用自身的安装/解压目录，正在被 Maple Tools 进程占用），点删除：
- 「移到回收站」失败弹窗 原始错误 = `SendToRecycleBin: PowerShell 结果解析失败`
- 「永久删除」失败弹窗 原始错误 = `Remove-Item: PowerShell 结果解析失败`

**根因方向**（待验证）：`src/main/ipc/fileExplorer.ts` 的 PS 子进程
始终 `exit 0` 然后通过 `MAPLE_RESULT_BEGIN/END` 包 JSON 给主进程解析
（refactor 见 3ad3e9e）。"结果解析失败"意味着 stdout 里没拿到那对
marker——可能是：
- PS 进程根本没跑到 marker 输出阶段（早期硬崩，比如缺 cmdlet）
- stdout 被混入 stderr / Write-Error 截断
- 在自我目录上 Remove-Item 时 PS 自身被卡死，子进程未正常结束

**修法思路**：
- 加详细 stdout / stderr 日志到 `formatPsDeleteError`，把 PS 真正输出
  的内容（前 N 行）拼到「原始错误」段，而不是只显示"结果解析失败"
- 极端情况（PS 卡死）下，从 markdown 解析失败的兜底分支里至少透传
  exit code 和 stderr 尾巴
- 操作自身安装目录的场景特殊处理——主进程检测到目标路径是
  `app.getPath('exe')` 的祖先时，直接拒绝并提示用户先关闭/移动 Maple
  Tools 自身

**为什么是这个方案**：弹窗里那串中文「PowerShell 结果解析失败」是
TS 侧 `parsePsDeleteResult` 找不到 marker 时的兜底文案，对用户没用。
真要修必须把 PS 的原始 stdout/stderr 暴露出来——这次的目标不是
"让所有删除一定成功"（那本来就不可能），而是失败原因可读。
