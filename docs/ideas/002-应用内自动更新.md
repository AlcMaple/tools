## 📖 详细说明

### electron-updater 接入 + CI workflow 上传 metadata

**目标效果**：发布 tag 后 GitHub Release 上同时挂出 `latest.yml` /
`latest-mac.yml` / `*.exe.blockmap` / `*.dmg.blockmap` 这些 electron-updater
要的元数据。安装 v0.2.0 的客户端能解析 `latest.yml` 知道有 v0.3.0 可用。

**已经做过的事**（避免重新踩坑）：
- 项目用 GitHub Releases 作为发布渠道（不是 S3 / 自建），所以 publish
  provider 用 `github`，token 部分由 electron-updater 自动从 release URL
  推断，CI 那边不用配 `GH_TOKEN`（只读 release 不要 token）
- v0.2.0 release 当前 5 个 asset（exe / dmg / zip × 2）—— 缺 metadata 文件
  是 workflow 里 `files:` glob 只匹配 `*.exe / *.dmg / *.zip` 没把 yml 和
  blockmap 带上。要在 `.github/workflows/release.yml` 的 `softprops/action-gh-release`
  的 `files:` 里追加 `artifacts/**/*.yml` 和 `artifacts/**/*.blockmap`
- electron-builder 会在 `dist/` 同时生成 `latest.yml`（Windows）/
  `latest-mac.yml`（macOS），blockmap 也自动产，不用手动配
- macOS 不签名时 electron-builder 会发警告但 latest-mac.yml 仍能生成,
  只是更新到 mac 客户端时系统会拒绝安装替换包——所以 mac 这边走另一条路

### 主进程 updater 模块

**目标效果**：
- App ready 后延迟 3 秒静默检查（不阻塞首屏）
- Windows: 走 `electron-updater` 的 autoUpdater，事件转发到渲染层
  （`update-available` / `download-progress` / `update-downloaded`）
- macOS: 走 GitHub REST API `/repos/:owner/:repo/releases/latest` 比对
  tag 和当前 `app.getVersion()`，发现更高版本就发 `update-available`
  事件给渲染层，附带 release HTML URL；不调用 autoUpdater
- IPC `updater:check`（手动触发）/ `updater:install`（用户确认后调用
  quitAndInstall）/ `updater:open-release-page`（mac 用，shell.openExternal）

**已经做过的事**（避免重新踩坑）：
- electron-updater 文档明确：macOS 上未签名 app 调用 quitAndInstall 会
  在 Sequoia 之后直接静默失败，更新文件落地但下次启动还是旧版（系统
  拒绝运行被替换的可执行文件）—— 不要尝试 mac 上调用 autoUpdater
- 项目 v0.2.0 release notes 写了"未做代码签名 / 公证"作为已知限制,
  本阶段不变更这条
- `app.isPackaged` 在 dev 模式下是 false，autoUpdater 在 dev 模式会
  抛 "Skipping update check because application is not packaged"——开发时
  无影响但要在 updater 模块入口处加 `if (!app.isPackaged) return`
  作为安全网

### 渲染层 updateStore + 顶部 banner

**目标效果**：
- banner 在所有页面顶部固定位置，能关闭（关掉后 session 内不再弹）
- 状态机：`idle → checking → available → downloading (Win only) → downloaded (Win only) | available-mac`
- Windows downloaded 状态：banner 显示「v0.3.0 已下载好，点击重启安装」
- macOS available 状态：banner 显示「v0.3.0 可用，点击前往下载」（跳浏览器）
- 检查中 / 失败 等状态不打扰用户（banner 不显示），仅手动按钮反馈

**已经做过的事**（避免重新踩坑）：
- 项目用纯 observable store 模式（参考 downloadStore.ts），不用 React
  Context / Zustand 之类。新建 `src/renderer/src/stores/updateStore.ts`
  跟现有风格保持一致
- 顶部布局已有 sticky 导航条，banner 应该插在导航条**下方**而不是上方
  （上方会和窗口标题栏挤），同时不能挡住任何页面内的 sticky bar
  （比如 Calendar 的周一-周日 chip）—— banner 放在导航条 sticky 容器里

### 设置页"关于"区块加「检查更新」

**目标效果**：设置页底部"关于"区块（应该已存在版本号显示）加一个按钮,
点击触发 `updater:check`，按钮状态根据 updateStore 显示文案：
「检查更新」/ 「检查中…」/ 「已是最新版本」/ 「v0.3.0 可用」

**已经做过的事**（避免重新踩坑）：
- 设置页 v0.2.0 重做过左栏导航结构，"关于"作为最后一个分类。新按钮
  应该加到这个分类里，不要新开 section
- 手动触发和自动触发走同一个 store action，区别只在 UI 反馈层
  （手动触发显示"已是最新版本"，自动触发同状态下静默不弹 banner）
