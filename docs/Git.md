
# 提交规范（个人用）

## 约定

| **类型**       | **代表含义**      | **适用场景 / 例子**                                          |
| -------------- | ----------------- | ------------------------------------------------------------ |
| **`feat`**     | **新功能**        | 为用户新增了某个可见或可用的功能。 *例：`feat: 新增激活状态下空格触发播放暂停`* |
| **`fix`**      | **修复 bug**      | 修复了代码库中的一个 Bug 或是报错。 *例：`fix: 修复点击提交按钮后页面崩溃的问题`* |
| **`docs`**     | **文档更新**      | 仅仅修改了文档，不涉及代码逻辑的改动。 *例：`docs: 更新 README 中的安装步骤`* |
| **`style`**    | **代码格式**      | 不影响代码实际运行逻辑的格式修改（如空格、缩进、补充缺失的分号等）。 *例：`style: 统一使用单引号格式化代码`* |
| **`refactor`** | **代码重构**      | 对代码结构进行调整或优化，既不是修复 Bug，也不是新增功能。 *例：`refactor: 提取播放控制逻辑为独立的公共组件`* |
| **`perf`**     | **性能优化**      | 旨在提升性能（如速度、内存占用等）的代码更改。 *例：`perf: 优化首页大图加载速度`* |
| **`test`**     | **测试**          | 添加缺失的测试代码，或更正现有的测试代码。 *例：`test: 新增用户登录接口的单元测试`* |
| **`build`**    | **构建系统/依赖** | 影响构建系统或外部依赖的更改（如 npm、webpack、Vite 相关的修改）。 *例：`build: 升级 React 版本至 18.0`* |
| **`ci`**       | **持续集成**      | 对 CI/CD 配置文件和脚本的更改（如 GitHub Actions、GitLab CI 等）。 *例：`ci: 新增自动部署到测试环境的流水线`* |
| **`chore`**    | **杂项/日常事务** | 不属于以上类型，且不修改源代码或测试文件的其他变更（如清理无用工具等）。 *例：`chore: 删除项目中无用的图片资源`* |
| **`revert`**   | **代码回滚**      | 撤销之前的某一次提交。 *例：`revert: 撤销 "feat: 新增搜索功能"`* |

- 完整格式：`类型(作用域): 简短描述` （`type(scope): subject`）
- 作用域的作用是额外说明这次提交具体影响了项目里的哪个模块、组件或范围
- 作用域是**可选的**。如果这次改动比较全局，或者实在归类不到某个特定的业务模块里，就可以直接省略括号
