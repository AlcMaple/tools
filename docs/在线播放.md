# 目标

- 快进时，中间不加载，比如从15 秒跳转到3 分 40 秒，15-3 分 40 秒这段时间不加载，从 3 分 40 开始加载
- 然后再从 3 分 40 跳转到 15 分，同理
- 如果此时回到 3 分 40，不存在加载，这个是已经加载完毕了的，直接零加载零延迟播放
- 如果此时继续回到 1 分 20 秒，由于 1 分 20 秒没加载过的，所以是要加载的

---

- 最终达到的效果是只要加载过的必定是 0 加载 0 延迟进行播放
- 如果关闭了应用、换一集看则清除缓存，下次如果再次看这一集的话全部重新加载
- 换一集看或者关闭应用会保留上次观看的进度，允许用户选择从上一次结束的地方继续看，或者重新看这一集
- 看完后不允许自动播放下一集，每次播放的时候需要用户点击开始播放才可以（虽然要用户点击开始播放才播放，但是用户没点击开始播放的时候已经可以先加载进度了，不需要用户点击开始播放的时候才加载）

# M3U8 (HLS 协议) - girigiri

- 直接使用第三方库 hls.js，当用户从 15 秒跳到 3 分 40 秒时，hls.js 在前端会自动计算出 3 分 40 秒对应的是第 44 个 .ts 文件。它会直接抛弃中间的请求，去请求 44.ts
- 当代理服务器收到前端请求 44.ts 时，主进程不仅去下载 44.ts，还立刻开启另外 3 个线程，去后台悄悄下载 45.ts, 46.ts, 47.ts 并存放在内存（Buffer）或本地临时目录中

# mp4 - xifan

- 当用户拖动进度条到 3 分 40 秒时，前端的 <video> 标签会立刻掐断当前的 HTTP 连接，然后向你的本地代理发送一个新的请求，请求头里会带上类似：Range: bytes=15000000-（意思是：给我从第 15MB 开始的数据）
- 代理服务收到前端带有 Range: bytes=X- 的请求
- 拿到真实视频的 Content-Length (总大小)
- 把后面的数据切成块（比如每块 5MB），开启 3 个 Node.js 线程去请求真实的视频服务器
    - 线程A 请求: Range: bytes=X 跑到 X+5MB
    - 线程B 请求: Range: bytes=X+5MB 跑到 X+10MB
    - 线程C 请求: Range: bytes=X+10MB 跑到 X+15MB
- 等线程A的数据拿到一点，就立刻通过 Node.js 的 pipe() 把数据流回写给前端浏览器。线程A下完了，接上线程B的数据继续喂
- 如果用户这个时候又跳转了，前端会主动 abort (断开) 连接。你的本地代理服务监听到 req.on('close') 事件后，必须立刻销毁当前正在后台下载的线程，释放资源，准备迎接下一次跳转请求
