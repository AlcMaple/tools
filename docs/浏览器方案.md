# cheerio 与 electron 与 LightPanda 方案

- 不要考虑外部的无头浏览器库
- 对于 Electron 来说，最好的“无头浏览器”就是它自己体内的那个 Chromium！

## 速度对比

Lightpanda 的宣传词是：“比 Chrome 快 11 倍，内存低 9 倍”，但这个对比是基于全功能的无头浏览器（Headless Browser）（比如配合 Puppeteer/Playwright 使用的 Chrome）而言的。

---

### 当前项目方案 (https/axios + cheerio) - 速度最快

- 原理：直接发起一个 HTTP 网络请求，拿到纯文本的一大串 HTML 字符，然后像字符串一样解析它
- 优点：不需要任何浏览器的“渲染（DOM/CSSCSSOM）”步骤，也不需要执行 JavaScript 引擎。它是纯内存中的文本分析
- 速度：只要网络不卡，几乎在毫秒级别完成

### Lightpanda - 速度第二

- 原理：虽然它是极速的 Zig 无头浏览器，但它毕竟是一个真实的浏览器，依然需要走复杂的网络协议栈、DOM 解析、构建树、跑一下 JavaScript 引擎并等待页面加载完成
- 速度：比下面的传统 Chrome 快很多，但无论如何也不可能战胜直接纯发 HTTP 请求的方案

### Electron 自带 Chromium - 速度第三

- electron 项目客户端它内部本身已经打包好了一个巨大的 Chromium 浏览器内核了。
- 如果用一个带 show: false 属性隐藏调用的 BrowserWindow 去抓取页面，它需要耗一定的内存去开一个新的进程进行解析

### 总结

抓取类似 bgm.tv 这种把关键信息直接渲染在返回 HTML 当中不需要额外跑复杂 JS 加载数据的网站，目前的 cheerio 解析已经是性能天花板

## 打包和分发

### 二进制问题

- Lightpanda 是用 Zig 写的 Native 二进制可执行文件
- 要支持 Windows(x64)、Mac(ARM 和 x64)，就必须把这三种架构的 Lightpanda 二进制包全部拖进项目中

### 打包问题

- 需要在electron-builder 繁琐配置 extraResources 或 extraFiles 去确保构建不同平台时复制正确的二进制文件
- 打包体积会变大
- mac 和 Windows 调用的方案不同
- 容易遇到权限拒绝（Permission Denied）、防病毒软件误杀拦截等一地鸡毛的问题
