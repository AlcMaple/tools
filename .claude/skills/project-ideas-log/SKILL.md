这个 skill 在每个项目根目录维护 `docs/ideas/` 文件夹，按 idea 阶段顺序累积编号 md 文件，每个文件是一个完整的开发周期记录。目标是让一周后回到项目时，10 分钟内恢复完整上下文。

## 核心原则

四条原则，**顺序即优先级**，后写的规则不能违反前面的：

1. **位置即状态** — 能用文件名/位置表达的，不开字段
2. **约定优于配置** — 一条规则覆盖所有情况，胜过逐项标注
3. **产物导向** — 描述「做完会得到什么」，不描述「要去做什么」
4. **未来的自己优先** — 文档为一周后的自己而写

每次想新增字段、段落、标记时先用这四条审一遍。

## 目录结构

```
项目根/
└── docs/
    └── ideas/
        ├── 001-初版MVP.md
        ├── 002-用户系统重构.md
        └── 003-Android适配.md   ← 最大序号即当前活跃
```

序号最大的文件 = 当前进行中，其他全是已完成。不创建 INDEX、不加状态字段、不加 emoji 前缀、不记录结束时间。

## 文件命名

格式：`NNN-简短主题.md`

- `NNN`：三位数字，从 001 开始，保证字典序等于时间序
- `简短主题`：3–8 个字，能一眼认出在做什么

不使用日期开头（idea 跨度可能从几周到几个月，日期会失真）。不加状态前缀（位置即状态）。

## 文件结构

每个文件包含四个段落，固定顺序：

```markdown
# NNN - 简短主题

> 开始：YYYY-MM-DD

## 🎯 这个阶段的产物

## 📋 TODO

## 📖 详细说明

## ✍️ 当前在做
```

### 🎯 这个阶段的产物

一段话，描述「这个阶段完成时，项目会得到什么」。

- ✅ "Biu 在 Android 上可安装运行，核心功能（播放/搜索/收藏）和 Mac 端体验一致；锁屏后台播放正常"
- ❌ "适配 Biu 到 Android 平台"（过程描述，不是产物）

产物描述可验证（达成则归档开新阶段），过程描述会无限拖延。

### 📋 TODO

动态清单，随时增删改。每条用一个标记开头：

| 标记 | 含义 |
|---|---|
| `[ ]` | 待做 |
| `[▶️]` | 进行中（同一时间最多一条） |
| `[x]` | 已完成 |
| `[🪟]` | 等 Windows 调试/验证 |
| `[⛔]` | 其他原因阻塞，简短写明原因 |

每条 todo 是一个功能/任务标题，不展开为多个子条目。需要细节去 📖 详细说明 写。

### 📖 详细说明

只对**大功能**或**有非显然决策**的 todo 展开，不需要每条 todo 都写。

每条详细说明的写法：

```markdown
### TODO 标题

**目标效果**：用户视角描述能感知到的效果，可量化更好。

**为什么是这个方案**：当时讨论的关键决策依据，淘汰了哪些备选。
```

不写技术实现细节（让代码自己说话）。重点写「为什么」，这是最容易遗忘也最贵的信息。

### ✍️ 当前在做

一两句话，指向 TODO 里 `[▶️]` 那一项的**具体下一动作**。

- ✅ "打开 player_native.ts:88，把 audioFocus 监听接进来"
- ❌ "继续做 Android 播放器适配"（无信息量）

单独成段而非挂在 todo 下，是因为这段是「恢复上下文的入口」，打开文件第一眼看到才有用。

## 维护规则

### 何时新建文件
当前文件「🎯 产物」描述的目标达成时，归档当前文件并新建序号 +1 的文件。

归档动作：
1. 把进行中的 `[▶️]` 改成 `[x]`
2. 清空「✍️ 当前在做」段落内容
3. 不改文件名，不加完成标记

### 何时不新建文件
当前阶段有新想法、要修 bug、临时插入小任务 → 加到当前文件 TODO。

判断标准：新事项是否服务于当前文件的「🎯 产物」描述？
- 是 → 加到当前 TODO
- 否 → 这条事项 hold 着，当前阶段完成后再开新文件

### 已完成的文件
- **不删** — 它们是项目的思考史
- **不改** — 除错别字外，归档后只读

## 反模式

以下设计不要使用：

- `INDEX.md` — 文件列表本身就是索引
- 状态字段 / 状态前缀 / emoji 状态标记 — 位置即状态
- 结束时间 — 下一个文件的开始时间即此文件的结束时间
- 单独的「下次开机第一件事」段落 — 已并入 ✍️ 当前在做
- 单独的「阻塞/卡点」段落 — 已并入 TODO 的 `[🪟]` `[⛔]` 标记
- 给每条 todo 都写详细说明 — 只对需要的写
- 在 TODO 里写技术实现步骤 — 颗粒度过细会变成假工作

## 初始化新项目

执行：

```bash
mkdir -p docs/ideas
```

在 `docs/ideas/001-<主题>.md` 中按下面的完整模板填写四段内容。

## 完整模板

```markdown
# 003 - Android 适配

> 开始：2026-05-08

## 🎯 这个阶段的产物
Biu 在 Android 上可安装运行，核心功能（播放/搜索/收藏）和 Mac 端
体验一致；锁屏后台播放正常。用户切到其他 App 时音乐不中断。

## 📋 TODO
- [x] Capacitor 项目骨架搭建
- [x] 平台抽象层迁移
- [▶️] Android 播放器内核适配
- [🪟] CapacitorHttp 在 Win 构建产物里的 CORS 行为验证
- [ ] 通知栏 MediaSession
- [ ] 深色模式适配
- [ ] Release 构建签名

## 📖 详细说明

### Android 播放器内核适配
**目标效果**：锁屏继续播放、通知栏可控制、切歌响应 < 200ms。
**为什么走原生桥**：Web Audio 在 Android 后台会被挂起，必须 MediaSession 保活。

### CapacitorHttp 验证
**为什么**：之前 fetch 在 Android WebView 的 CORS 行为和 Mac 不一致，
迁到 CapacitorHttp 走原生网络栈，但 Win 端的构建产物没验证过。

## ✍️ 当前在做
打开 player_native.ts:88，把 audioFocus 监听接进来。
完成后跑 `npm run android:dev` 看锁屏控制能不能弹出。
```
