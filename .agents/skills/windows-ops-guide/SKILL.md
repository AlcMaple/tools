description: 当某项工作需要用户在 Windows 机器上手动执行（命令、火绒/杀软操作、系统设置、文件检查等），而对话窗口只在另一台机器（如 macOS）上看得到时，把操作步骤整理成一份独立的 md 指导文件写进仓库，让用户切到 Windows 后照着做、回填结果。适用于安全排查、环境配置、打包部署等跨机器协作场景。触发词：「需要在 Windows 操作」「我在 Windows 看不到这个窗口」「写个 md 指导我」「整理成文件」。

这个 skill 解决一个具体痛点：**对话发生在 A 机器（如开发用的 Mac），但实际操作要在 B 机器（Windows）上做，用户切过去就看不到聊天记录了。** 口头/聊天里的步骤一离开窗口就丢，所以要落成仓库里的 md 文件，让用户在 Windows 上打开照做。

## 何时用

- 用户明确说「需要在 Windows 上操作 / 我那边看不到这个对话 / 写成 md 指导我」。
- 你正在给出一串需要在另一台机器执行的命令或 GUI 操作（杀软查杀、注册表/计划任务检查、conda 环境重建、打包传输、驱动/CUDA 安装等），而那台机器不是当前对话所在的机器。

不用于：操作就在当前机器、或几条命令当场跑完即可的情况——那种直接在对话里给就行，不必造文件。

## 产物

仓库 `docs/` 下一份独立 md（命名见下），用户在 Windows 上打开即可独立执行，**不依赖任何聊天上下文**。做完每步把结果填回文件，发回来给你判断下一步。

## 命名

`docs/<主题>-<用途>指南.md`，例如：
- `安全排查-清除后复查指南.md`
- `环境配置-CUDA安装指南.md`

主题用中文短词，能一眼认出在解决什么。若是某个既有排查/任务文档的延续，主题词保持一致（同一件事的文件聚在一起）。

## 编写规则（顺序即优先级）

1. **自包含** — 文件里必须复述清楚背景：现在处于什么状态、为什么要做这些、上一步结论是什么。假设读者**没有**聊天记录，只有这一个文件。

2. **环境前提写在最顶上** — 用什么终端（cmd 还是 PowerShell，二者命令常不通用）、路径占位怎么替换（如「把 `obsidian` 换成你实际的环境名」）、需要管理员权限吗。这些一错全错，必须开篇点明。

3. **可勾选 + 可回填** — 步骤用 `- [ ]` 复选框；每一步末尾留一个「📝 结果」代码块占位，让用户把命令输出/看到的现象直接贴进去。文件因此同时是「操作清单」和「结果记录单」。

4. **命令可直接复制** — 用 ```bat / ```bash 代码块，一条命令一行，行内用 `::`（cmd）或 `#` 注释说明每条在干嘛。不要把多条揉成一行让用户去拆。

5. **判断标准随命令给** — 每条命令后面紧跟「有输出 = …，无输出 = …」式的判读规则，让用户自己就能初判，而不是所有结果都攒着等你看。

6. **危险操作设护栏** — 删注册表项、删计划任务、`conda env remove`、`rm -rf` 这类不可逆操作，明确写「先记下内容发回来确认，别自己删」或给出确认前提。

7. **按优先级排序 + 标关键步** — 最该先做、最能定位问题的步骤排最前并标「最关键，先做这个」。可顺带做的排后面。

8. **收尾留汇总区** — 文件末尾一个「做完汇总」段落，列出要回报的几项，方便用户一次性发回、你一次性判断。

## 写完之后

- 告诉用户文件路径，简述每一步在做什么。
- 若这份指南是某个既有文档（如排查记录）的延续，**提议**把进度回填到原文档的结果区，保持单一事实源。
- 不替用户在 Windows 上执行——你在另一台机器上，执行和回填由用户完成。
